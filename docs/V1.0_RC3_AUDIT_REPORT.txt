Bellwether v1.0 RC3 — Postgame and Async Runtime Audit

Scope
- Implemented the agreed RC3 postgame milestone.
- Added independent safe harvesting of completed background AI jobs.
- Added inspectable queue/job lifecycle logs and explicit job states.
- Preserved one-inference-at-a-time low-memory architecture.
- Simplified README.md.

Postgame
- All six canonical ending families activate a persistent postgame state.
- Existing ordinary gameplay actions remain available after resolution.
- Ending-specific village conditions are persisted.
- Continued community life, home life and side-mystery follow-up are tracked.
- Farming, jobs, relationships, hobbies, economy, travel and ordinary life remain the existing authoritative systems rather than being replaced by a separate postgame simulation.

Async runtime corrections
- Queue status distinguishes queued, running and completed-awaiting-validation jobs.
- Job list exposes id, key, kind, revision, signature, age and running time.
- Recent event log records queued, coalesced, started, finished, failed, harvested, applied and rejected/stale outcomes.
- A daemon result-harvester checks for completed work and applies results under the same game lock used by player actions.
- Background worker still never mutates game state directly.
- Director batches older than 8 simulation revisions are rejected as stale.
- Historical queued_or_running status key was preserved after regression testing caught its accidental removal.
