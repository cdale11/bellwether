BELLWETHER GAME — DESIGN CONSTITUTION Version 1.0 Date: 7 July 2026

PURPOSE

This document defines the creative and systemic identity of Bellwether.
It is intended to guide all development after the frozen Bellwether
v0.1.0 baseline.

The Constitution does not replace technical specifications,
implementation plans, story bibles, or test plans. Its purpose is to
answer a more fundamental question:

What kind of game is Bellwether supposed to become?

When future features, optimizations, narrative ideas, or architectural
changes conflict with this Constitution, development should stop and
reconsider the design before proceeding.

1.  THE CORE IDENTITY OF BELLWETHER

Bellwether is a persistent, state-driven village life RPG that begins as
a compelling ordinary-life simulation and gradually becomes a dangerous
psychological and supernatural horror experience.

Bellwether is not primarily a linear horror story with life-simulation
decoration.

Bellwether is not primarily a farming game with occasional scripted
scares.

Bellwether is not a sandbox in which procedural generation is allowed to
replace authored meaning.

Bellwether is a living fictional world in which ordinary life,
relationships, work, ecology, weather, psychology, investigation,
supernatural forces, and authored mysteries interact through shared
state.

The player should want to return because Bellwether feels like a place
that continues to change, where actions matter, where routines become
personally meaningful, and where the world affects the player as
strongly as the player affects the world.

2.  THE FROZEN FOUNDATION

Bellwether v0.1.0 is the frozen stable baseline.

All post-v0.1.0 development must preserve the working behavior of
established systems, including:

-   living village simulation;
-   NPC movement and activity;
-   traffic and transport behavior;
-   Director scheduling and concurrency protection;
-   seasons, weather, temperature, and dayparts;
-   authored story conversations;
-   bounded LLM-driven ordinary conversations;
-   conversation continuity;
-   social consequences;
-   relationship privacy;
-   social memory;
-   accumulated player and game context;
-   story integration;
-   controlled horror escalation;
-   psychological consequences;
-   branches;
-   failure states;
-   recovery paths;
-   endgame and reachable endings;
-   save/load persistence;
-   migration behavior;
-   release diagnostics;
-   executable permissions.

New development should be additive wherever possible.

Existing working systems must not be casually rewritten, simplified,
bypassed, or removed. Architectural migration is permitted only when
there is a clear need, an explicit migration plan, and strong regression
testing.

3.  ORDINARY LIFE MUST BE WORTH LIVING

The ordinary-life game must be enjoyable even if the main mystery is
temporarily ignored.

A player should be able to spend substantial in-game time:

-   gardening;
-   growing edible vegetables;
-   cooking;
-   maintaining and repairing a home;
-   working;
-   earning and spending money;
-   shopping;
-   using village services;
-   walking;
-   exploring;
-   visiting people;
-   attending events;
-   caring for possessions;
-   observing weather and seasonal change;
-   participating in village routines;
-   developing relationships.

These systems must not exist merely to increase hidden prerequisite
counters.

Ordinary life must have intrinsic value.

The player should develop attachments to routines, places, people,
possessions, animals, plants, and expectations. Horror becomes stronger
when it threatens, corrupts, or contradicts things the player genuinely
understands and values.

4.  ACTIVITIES MUST BE SYSTEMS, NOT BUTTONS

Major activities should have meaningful internal structure.

Gardening must not be reduced to:

Click “Garden” → receive vegetables.

A gardening system may involve:

-   choosing crops;
-   seasonal suitability;
-   planting;
-   soil or plot condition;
-   watering;
-   rainfall;
-   temperature;
-   growth stages;
-   tending;
-   pests or disease where appropriate;
-   harvesting;
-   storage;
-   cooking;
-   selling;
-   gifting;
-   sharing;
-   failure and recovery.

Likewise, cooking, repairing, working, farming, and other recurring
activities should create decisions, anticipation, consequences, and
connections with other systems.

The target is medium-effort progression: substantial enough to create
investment and satisfaction, but not an extreme MMORPG-style grind.

Repetition should produce progression, mastery, resources,
relationships, knowledge, or changing opportunities rather than
meaningless time consumption.

5.  PLAYER FREEDOM

The player should be free to live in Bellwether according to their own
priorities.

The game should not force constant pursuit of the main story.

A player may choose to:

-   work;
-   garden;
-   cook;
-   socialize;
-   investigate;
-   explore;
-   improve their living situation;
-   observe unusual events;
-   ignore warnings;
-   pursue relationships;
-   become isolated;
-   follow routines;
-   deliberately break routines.

The world, however, does not wait for the player.

NPC lives continue.

Weather changes.

Seasons progress.

Relationships evolve.

Information spreads.

Events occur.

Opportunities can appear and disappear.

Supernatural activity may occur whether or not the player is
investigating it.

Freedom does not mean a static world. It means freedom inside a world
that has its own momentum.

6.  THE WORLD MUST BE CAUSAL

Bellwether should increasingly operate through interconnected causal
systems.

The long-term causal structure should resemble:

environment → ecology → needs → intentions → movement → encounters →
information exchange → knowledge changes → dialogue changes →
relationship changes → psychological changes → future decisions →
village consequences → story opportunities → supernatural opportunities
→ altered environment

This chain is conceptual, not a requirement that every event pass
through every stage.

The important principle is that systems should increasingly affect one
another.

Weather should affect plants, animals, travel, activities, mood,
schedules, and events.

Schedules should affect meetings.

Meetings should affect knowledge.

Knowledge should affect dialogue and decisions.

Relationships should affect behavior.

Economic circumstances should affect choices.

Player actions should create consequences.

Supernatural forces should exploit and corrupt existing systems rather
than existing in a separate horror minigame.

7.  THE VILLAGE IS A CHARACTER

Bellwether itself should feel alive.

The village is not merely a menu of locations.

Its identity emerges from:

-   geography;
-   districts;
-   paths;
-   roads;
-   interiors;
-   businesses;
-   homes;
-   public spaces;
-   private spaces;
-   opening hours;
-   transport;
-   weather exposure;
-   vegetation;
-   animals;
-   seasonal accessibility;
-   social habits;
-   local events;
-   rumors;
-   histories;
-   mysteries;
-   supernatural influence.

Places should acquire meaning through repeated use.

A bus stop matters because the player has waited there in rain, met
people there, learned the timetable, missed the last bus, and perhaps
later witnessed a bus arrive when no bus should exist.

A garden matters because the player planted it, tended it, watched
weather affect it, cooked from it, and may later discover that something
impossible has grown there.

Familiarity is a gameplay resource and a horror resource.

8.  THE ENVIRONMENT IS SIMULATED, NOT DECORATIVE

Animals, vegetation, weather, seasons, temperature, and other
environmental elements should respond to shared environmental state.

The environment should include appropriate autonomous behavior.

Examples include:

-   plants growing according to season and conditions;
-   vegetation changing through the year;
-   animals appearing or behaving differently according to weather,
    time, food availability, disturbance, and season;
-   birds becoming quiet under unusual conditions;
-   insects appearing seasonally;
-   paths becoming difficult under certain conditions;
-   gardens responding to rainfall and temperature;
-   wildlife adapting to ordinary environmental cues.

Supernatural forces may later manipulate these same systems.

The supernatural should therefore be able to produce meaningful
ecological contradictions:

-   unseasonal flowering;
-   impossible animal behavior;
-   localized weather;
-   sudden silence;
-   vegetation patterns that form messages;
-   repeated animal movements;
-   ecological changes remembered differently by different residents.

Such effects should be stateful and consequential, not merely random
spooky prose.

9.  NPCS HAVE LIVES BEYOND THE PLAYER

Bellwether should use a layered population model.

A relatively small core cast should receive deep characterization,
personal histories, relationships, needs, memories, psychological
development, and authored arcs.

A larger background population may provide village density and social
texture without requiring equivalent narrative depth.

Important NPCs should have lives independent of the player.

They may:

-   work;
-   rest;
-   travel;
-   shop;
-   eat;
-   visit;
-   argue;
-   reconcile;
-   form attachments;
-   lose trust;
-   become ill;
-   change routines;
-   exchange information;
-   misunderstand information;
-   develop fears;
-   react psychologically to the village;
-   make consequential choices.

Important events may occur without the player witnessing them.

The player may discover causes, fragments, rumors, consequences, or
conflicting accounts later.

NPC autonomy must remain bounded by authored identity, world rules, and
validated state transitions.

10. PURPOSE-DRIVEN BEHAVIOR

NPC movement should evolve beyond simple destination assignment.

The future model should support concepts such as:

need: obtain food intent: buy breakfast possible destinations: bakery,
village shop, home constraints: opening hours, money, weather,
obligations, distance, preferences, relationships, fear, current
knowledge chosen destination: bakery

Directors and LLMs may help choose among bounded plausible
possibilities.

They must not invent unrestricted locations, overwrite canon, or create
impossible transitions.

The goal is not unrestricted artificial intelligence.

The goal is believable bounded agency inside a coherent simulation.

11. THE PLAYER CHARACTER EVOLVES

The player character is neither completely blank nor completely fixed.

Bellwether should provide enough authored identity and circumstance to
create narrative coherence while allowing the character to be shaped
substantially through play.

The character should evolve according to:

-   repeated behavior;
-   relationships;
-   choices;
-   habits;
-   failures;
-   successes;
-   investigation style;
-   moral decisions;
-   fears;
-   exposure to supernatural forces;
-   psychological consequences;
-   memories from the current and, eventually, previous runs.

The town should react to who the player becomes.

Different villagers may perceive the same player differently.

The player’s behavioral history should matter beyond simple numerical
reputation.

12. PSYCHOLOGY IS PART OF THE WORLD MODEL

Bellwether’s supernatural forces can affect psychology.

The village may influence:

-   perception;
-   memory;
-   trust;
-   fear;
-   attachment;
-   avoidance;
-   obsession;
-   denial;
-   interpretation of events.

Psychological state should not simply be a single sanity bar.

Where practical, psychological consequences should manifest through
behavior, perception, options, memory, relationships, interpretation,
and systemic effects.

The player and NPCs may both be affected.

However, the game must preserve enough authoritative state to
distinguish deliberate supernatural contradiction from accidental LLM
hallucination or simulation error.

13. ECONOMIC PRESSURE SHOULD MATTER

Bellwether should eventually contain a meaningful but moderate economy.

Money and resources should create pressure without turning the game into
constant financial punishment.

Possible pressures include:

-   food;
-   home maintenance;
-   heating or seasonal costs;
-   transport;
-   tools;
-   seeds and supplies;
-   repairs;
-   services;
-   optional comforts;
-   social expenses;
-   emergencies.

Jobs and livelihoods should interact with:

-   schedules;
-   skills;
-   relationships;
-   weather;
-   seasons;
-   businesses;
-   events;
-   story circumstances;
-   psychological state;
-   supernatural disruption.

Economic systems must not be implemented as isolated number generators.

They should create reasons to participate in village life.

14. DANGER IS REAL

Bellwether eventually becomes very dangerous.

The player can die.

A player death may end the current run.

NPCs may face severe consequences, including disappearance or death
where appropriate to authored and systemic design.

Danger should emerge from understandable causal circumstances whenever
possible.

Deaths and failures should not feel like arbitrary random punishment.

Bellwether should also preserve meaningful failure states and recovery
paths where appropriate.

Not every mistake must be fatal, but the world must be capable of
lasting consequences.

15. BELLWETHER DOES NOT WANT PEOPLE TO LEAVE

A central horror principle is that Bellwether is difficult or impossible
to escape in a deeper sense.

The town and its supernatural forces may manipulate residents
psychologically and systemically so that leaving becomes difficult,
self-defeating, forgotten, rationalized away, interrupted, or otherwise
prevented.

This should not be represented only by an invisible wall or a message
saying that the player cannot leave.

The inability to escape should emerge through layered narrative and
systemic mechanisms.

Residents may normalize contradictions.

Plans may change.

Memories may shift.

Relationships may become anchors.

Routes may behave strangely.

Circumstances may repeatedly intervene.

Different characters may experience or explain this imprisonment
differently.

The exact mechanisms should be developed through authored mysteries and
systemic horror design.

16. HORROR THROUGH LEARNED NORMALITY

Bellwether’s central horror philosophy is:

First teach the player what normal means.

Then violate it.

The game should establish understandable patterns:

-   NPC routines;
-   transport schedules;
-   seasonal behavior;
-   weather patterns;
-   relationships;
-   opening hours;
-   animal behavior;
-   vegetation;
-   architecture;
-   local customs;
-   memories;
-   conversation history.

Only after familiarity develops should supernatural forces begin to
contradict, corrupt, imitate, rearrange, or weaponize these patterns.

Examples may include:

-   impossible NPC locations;
-   contradictory bus appearances;
-   conversations remembered differently;
-   writing or symbols appearing on buildings;
-   localized or impossible weather;
-   season changes that should not occur;
-   plants behaving impossibly;
-   animals following unnatural patterns;
-   buildings appearing subtly altered;
-   residents remembering events that did not occur;
-   the player’s own prior words being changed or reflected back.

The horror should be creative, systemic, and state-aware.

17. MULTIPLE MYSTERIES, EVENTUAL CONNECTION

Bellwether should contain multiple mysteries.

Initially, they may appear unrelated.

Different players may encounter them in different orders.

Some mysteries may be discovered through:

-   relationships;
-   work;
-   exploration;
-   archives;
-   environmental observation;
-   transport;
-   village rumors;
-   repeated routines;
-   supernatural incidents;
-   previous-run memory.

Over time, major mysteries should increasingly connect.

The game should ultimately explain approximately 90–95 percent of its
central supernatural structure in the most revealing paths, while
retaining a small residue of genuine uncertainty.

Mystery must not become incoherence.

The player should be able to form theories, test them, discover
evidence, revise conclusions, and eventually understand most of what
Bellwether is and why events occur.

18. AUTHORED CANON AND GENERATIVE REACTIVITY

Bellwether uses a hybrid architecture.

Authored content controls:

-   canon;
-   major mysteries;
-   important facts;
-   revelations;
-   major story beats;
-   key story conversations;
-   personal arcs;
-   branches;
-   major failures;
-   recovery structures;
-   endings.

LLMs and Directors may support:

-   ordinary conversation;
-   bounded interpretation;
-   social inference;
-   purpose selection;
-   contextual reactions;
-   bounded variation;
-   simulation coordination;
-   world commentary;
-   non-canonical texture.

The use of LLMs and Directors should expand where they improve
reactivity and variation.

They must remain constrained by authoritative state, validation, and
canon.

The objective is emergent variation without narrative incoherence.

19. NO TWO RUNS SHOULD BE PERFECTLY DETERMINISTIC

Bellwether should allow natural variation between runs.

Variation may emerge from:

-   weather;
-   seasons;
-   schedules;
-   NPC decisions;
-   meetings;
-   information flow;
-   relationship development;
-   player behavior;
-   ecological conditions;
-   Director choices;
-   bounded LLM interpretation;
-   event timing;
-   supernatural targeting and response.

The game should retain authored mysteries and coherent canon.

Variation should alter the route through the world and the consequences
of interaction, not randomly rewrite the truth.

20. DEATH AND CROSS-RUN MEMORY

Bellwether should eventually support multi-run architecture.

A run may end through death or other terminal outcomes.

A new run should not necessarily be a complete erasure.

Future architecture may allow memories, impressions, fragments,
instincts, anomalies, or other carefully designed state to persist
across runs.

The intended emotional model is recurrence with accumulating
significance.

Cross-run persistence should deepen mystery and character development
rather than simply function as a conventional stat-based New Game Plus
mode.

Different entities may remember different amounts.

Memory may be incomplete, distorted, asymmetric, or dangerous.

This architecture must be designed carefully in a later dedicated
development phase and must not be prematurely bolted onto unrelated
systems.

21. THE PRIMARY PLAYER FANTASY

The central player fantasy is:

I have entered a fictional world that keeps changing. I can build a life
there. My actions affect the world, and the world changes me.

The player should feel:

-   curiosity about what will happen next;
-   attachment to ordinary routines;
-   investment in personal progress;
-   concern for residents;
-   interest in changing relationships;
-   anticipation of seasonal and village events;
-   satisfaction from mastering activities;
-   uncertainty about autonomous events;
-   fear when familiar systems become unreliable;
-   a desire to understand the connected mysteries;
-   awareness that choices have consequences.

The game should produce stories that arise partly from authored design
and partly from the interaction of systems.

22. PROGRESSION PHILOSOPHY

Progression should be substantial and satisfying.

The player should experience medium-effort goals rather than instant
rewards or extreme grinding.

Progression may include:

-   skills;
-   tools;
-   home improvements;
-   garden development;
-   recipes;
-   resources;
-   employment opportunities;
-   access to places;
-   trust;
-   relationships;
-   social knowledge;
-   investigation knowledge;
-   practical knowledge of village routines;
-   psychological change;
-   cross-run understanding.

Progression systems should connect to one another where appropriate.

A relationship may create a work opportunity.

A job may expose the player to a location at unusual hours.

A location may reveal an ecological anomaly.

An anomaly may connect to a mystery.

A mystery may alter a relationship.

A supernatural event may destroy or transform an ordinary source of
stability.

23. PART 1 MUST BUILD A SHARED WORLD SUBSTRATE

The immediate post-v0.1.0 development stage is Part 1: World Expansion
Architecture.

Part 1 must not merely add arbitrary locations.

It should establish a coherent, data-driven world model capable of
supporting future:

-   districts;
-   locations;
-   interiors and exteriors;
-   adjacency;
-   route choice;
-   travel time;
-   transport connections;
-   opening hours;
-   ownership;
-   access rules;
-   public and private space;
-   weather exposure;
-   seasonal accessibility;
-   NPC location suitability;
-   purpose-driven movement;
-   businesses;
-   services;
-   jobs;
-   economic activity;
-   player activities;
-   social encounters;
-   ecological state;
-   story staging;
-   investigation;
-   supernatural modification.

Part 1 should be implemented as a narrow, tested extension of the frozen
baseline.

It should prove the architecture with a small playable vertical slice
before large-scale content expansion.

24. DEVELOPMENT PRINCIPLE: DEPTH BEFORE BREADTH

Do not add twenty shallow locations when five deeply connected locations
can prove the model.

Do not add ten jobs if none create interesting schedules, relationships,
economic choices, or story opportunities.

Do not add animals as decorative text if they cannot respond to
environmental state.

Do not add farming if crops are merely timers with no connection to
weather, seasons, cooking, economy, or relationships.

Do not add procedural events merely to claim replayability.

Each major system should justify itself through meaningful interaction
with the rest of Bellwether.

25. TESTING IS A DESIGN ACTIVITY

Technical correctness is necessary but insufficient.

Every major build must be tested from the player’s perspective.

Testing should include:

-   compilation;
-   focused automated tests;
-   structural simulation regression;
-   save/load compatibility;
-   migration testing where needed;
-   scheduler unlock verification;
-   loading-state recovery;
-   Qwen fallback behavior;
-   authored conversation preservation;
-   free-text conversation behavior;
-   relationship and memory persistence;
-   weather and temperature progression;
-   transition validation;
-   executable permissions.

Major gameplay changes additionally require player-perspective
playthrough testing.

The tester should read prose and options sequentially without relying on
code knowledge and ask:

-   Is the world understandable?
-   Is ordinary life enjoyable?
-   Is progression satisfying?
-   Is the grind meaningful rather than excessive?
-   Are objectives discoverable?
-   Do activities have enough depth?
-   Do NPCs appear to live independently?
-   Do environmental systems visibly matter?
-   Are choices consequential?
-   Are relationships understandable?
-   Does investigation make sense?
-   Are failures comprehensible?
-   Are recovery paths discoverable?
-   Does horror emerge from learned normality?
-   Do endings follow causally from player behavior?
-   Does the village feel like a place rather than a collection of
    features?

A system that passes unit tests but produces a poor player experience is
not finished.

26. FINAL DEVELOPMENT RULES

All future Bellwether development should obey the following principles:

1.  Preserve the frozen v0.1.0 baseline.
2.  Build ordinary life deeply enough to be enjoyable without constant
    story pressure.
3.  Give the player freedom while allowing the world to continue
    independently.
4.  Connect systems through shared authoritative state.
5.  Make activities involved and consequential rather than single-click
    abstractions.
6.  Keep economic pressure meaningful but moderate.
7.  Give core NPCs autonomous lives beyond the player.
8.  Use a layered population model with a deep core cast and lighter
    background residents.
9.  Allow the player character to evolve through behavior and
    consequences.
10. Let the village and supernatural forces affect psychology.
11. Simulate environmental response through weather, season,
    temperature, vegetation, and animals.
12. Establish normal patterns before corrupting them for horror.
13. Make danger real and permit terminal run outcomes.
14. Treat Bellwether’s inability to release its residents as a central
    systemic and narrative horror principle.
15. Build multiple mysteries that increasingly connect.
16. Explain most of the central truth while preserving a small residue
    of uncertainty.
17. Expand LLM and Director use through bounded, validated reactivity
    rather than unrestricted authorship.
18. Permit natural variation without sacrificing canon.
19. Design future cross-run memory as a meaningful narrative system, not
    a superficial New Game Plus.
20. Prefer depth, causality, and interconnection over feature count.
21. Test major features through genuine player-perspective play.
22. Remember the primary fantasy: the player enters a changing fictional
    world, affects it, and is changed by it.

CLOSING STATEMENT

Bellwether should become a place the player learns to live in before
they fully understand why they cannot leave.

Its ordinary systems should create attachment.

Its autonomous systems should create surprise.

Its relationships should create responsibility.

Its mysteries should create curiosity.

Its progression should create investment.

Its environment should create a sense of place.

Its supernatural forces should understand enough about the world to
violate its rules deliberately.

Its recurring structure should allow knowledge, memory, and consequence
to accumulate across experiences.

The ultimate measure of Bellwether is not the number of systems it
contains.

The measure is whether those systems combine to make the player feel
that Bellwether is alive, that it remembers, that it changes, and that
entering it again will never produce exactly the same life.

BELLWETHER — COMPREHENSIVE PROJECT CONTEXT AND DEVELOPMENT BRIEF
Version: Living Vision Document Purpose: Authoritative context for
future ChatGPT, coding-agent, design, writing, and engineering
conversations.

====================================================================== 1.
PROJECT IDENTITY
======================================================================

Bellwether is a hybrid narrative village life-sim, RPG, interactive
fiction game, visual-novel-style character drama, systemic simulation,
mystery, and psychological/social horror game.

The player moves into and lives in Bellwether, a contemporary rural
English village and its immediate surroundings. The game should
initially be inviting, comforting, curious, and rich with ordinary life.
The village must feel alive: residents have routines, goals,
relationships, conversations, memories, conflicts, and lives that
continue during simulated in-game time even when the player is
elsewhere.

The supernatural premise is not merely a horror story layered over a
life-sim. The simulation itself is the medium through which horror
emerges. The town is, in some sense, alive. It affects the psychology
and behaviour of its residents and the player; residents and the player,
in turn, affect the town. The exact mechanism must eventually have an
authored explanation that accounts for roughly 95% of what happens,
while retaining a small residue of genuine ambiguity.

Bellwether must remain playable as a life-sim/RPG even after the
authored central mystery reaches a true ending or one of multiple fixed
major endings.

Core reference points: - Stardew Valley: freedom, ordinary routines,
progression, gardening/farming, relationships, economy, reasons to
return. - The Sims: a sense of autonomous lives and social systems, but
Bellwether should not become a detailed needs-management simulator. -
FROM (TV series, with Boyd, Tabitha, Jade, etc.): living
trapped-community mystery, competing theories, hostile unknowability,
interlocking mysteries, visions, impossible or shifting geography, and
the social consequences of prolonged fear. - Higurashi: attachment to
ordinary village life, recurrence, contradictions between cycles,
unreliable interpretation, social paranoia, layered causes, and horror
that becomes stronger because the player first cares about normal
life. - Disco Elysium: psychologically expressive RPG systems, internal
subjectivity, skill-like voices or interpretations, failure that can be
narratively interesting, and a player character shaped through
choices. - Doki Doki Literature Club: selective visual/interface/reality
violations and the possibility that the game presentation itself can
become unreliable, while Bellwether must remain playable unless a
genuine failure state is triggered.

These are tonal and systemic references, not templates to copy.

======================================================================
2. NON-NEGOTIABLE DEVELOPMENT RULES
======================================================================

Bellwether already has a materialized, working codebase with many
existing systems. Existing features that work are valuable and must be
preserved.

Future development must be ADDITIVE and CONSERVATIVE: - Do not casually
rewrite, remove, simplify, replace, or overwrite working systems. - Do
not regress existing simulation, conversation, persistence, Director,
weather/season, story, horror, relationship, investigation, branch,
recovery, endgame, testing, UI, or executable behaviour. - Before
changing an existing subsystem, inspect how it currently works. - Extend
existing architecture where possible rather than creating duplicate
parallel systems. - If a working feature appears to conflict with a new
request, consult the user before destructive architectural changes. -
Preserve save compatibility wherever reasonably possible; use
migration/defaulting for new state. - New systems must integrate with
authoritative existing state rather than inventing duplicate truth. -
Run existing diagnostics and targeted tests after changes. - Do not
claim a build is complete merely because code compiles. Playability,
integration, presentation, and regression safety matter. - The user does
not want to edit source files manually. Development deliverables should
be finished runnable builds, complete replacement files,
patches/bundles, or packaged projects—not instructions requiring manual
code editing. - UI must not become cluttered. The game should play
smoothly and remain inviting to return to.

The project’s design and documentation files are specifications and
should be treated as part of the authoritative context.

======================================================================
3. PLAYER EXPERIENCE AND AUTONOMY
======================================================================

The player should have broad autonomy. On an ordinary in-game week, the
player may choose among: - talking to villagers, - pursuing friendships
or romance, - working flexible jobs or long-term employment, - gardening
and eventually farming, - cooking and eating, - buying and selling
goods, - restoring and decorating the cottage, - exploring, - fishing,
reading, collecting, foraging, crafting, local history, photography,
birdwatching, music, volunteering, pub activities, pets, or other
suitable hobbies, - investigating mysteries, - following authored story
missions, - engaging with procedural errands and social situations, - or
simply living an ordinary village life.

The player must be able to ignore the central story for long periods.
The game may remind or tempt the player through corruption, revelations,
anomalies, consequences, rumours, and pressure around unresolved
mysteries, but it should not force constant linear story progression.

Story investigation may become pressured at appropriate points,
especially when the town’s condition worsens, but player freedom remains
a core value.

In-game days are variable in real-world duration. The player decides how
much to do. Time advances through actions and simulated in-game time,
not real-world elapsed time while the application is closed.

After roughly the first 3–5 hours, the desired feelings are: -
comfort, - curiosity, - attachment, - the sense that the town is alive
and changing, - awareness that NPCs interact with each other and alter
state, - and a subtle feeling that something may be wrong.

The player should always have reasons to want to return to Bellwether.

======================================================================
4. ORDINARY LIFE AS A FULL GAME, NOT FILLER
======================================================================

The peaceful life-sim is not a waiting room before horror. It must be
enjoyable enough to sustain play on its own.

Long-term peaceful motivations may include all of the following: -
cottage restoration, - relationships and romance, - careers and jobs, -
becoming accepted in the community, - gardening, - cooking, - economy
and profit, - exploration, - skills, - hobbies, - collecting, -
farming, - local knowledge, - social reputation, - personal identity
formation.

Mechanical depth should approach games such as Stardew Valley, but be
somewhat simpler and better suited to a text-heavy interface. Systems
should have meaningful interactions without excessive repetitive
micromanagement.

======================================================================
5. COTTAGE, GARDENING, FARMING, COOKING, AND ECONOMY
======================================================================

COTTAGE: The cottage can become a major progression anchor. Possible
systems include: - repairing rooms, - improving facilities, - buying
furniture, - decorating, - expanding storage, - unlocking or upgrading
cooking facilities, - improving the garden, - visible persistent
changes, - NPC reactions to the player’s home and choices.

GARDENING: Gardening should be mechanically meaningful: - crops, -
vegetables, - herbs, - flowers, - seasonal considerations, - planting
and harvest, - produce used for cooking, - produce eaten by the
player, - produce sold for profit, - gifting and social use where
appropriate.

The text-based format should reduce tedious friction compared with a
full graphical farming sim.

FARMING: Very late in progression, the player may acquire larger
agricultural land and effectively become a farmer. This should require
substantial money and progression and should connect to the proper town
economy.

COOKING AND FOOD: Ingredients and recipes should matter. Cooking may
support: - health and recovery, - eating, - gifts, - social events, -
hobbies, - economy, - possible business opportunities, - survival or
preparation during dangerous phases.

ECONOMY: Bellwether should have a proper town economy. Money is
important but should not be brutally difficult to obtain. Some
RPG/MMORPG-like grind is acceptable and desired, provided it is
rewarding and not oppressive. The player can: - earn wages, - complete
side jobs, - sell produce and crafted/foraged goods, - make profits, -
spend on food, repairs, seeds, tools, travel, gifts, leisure, upgrades,
and eventually land.

Economic state should be able to change dynamically.

======================================================================
6. JOBS, SKILLS, HEALTH, AND RPG PROGRESSION
======================================================================

JOBS: Support both: - flexible side jobs and quest-like work, -
longer-term daily or scheduled employment.

Potential workplaces include: - bakery, - village shop, - pub, -
farms, - clinic, - council, - and other suitable institutions.

Jobs may: - have schedules, - progress, - be lost, - create
relationships and reputation, - open or close opportunities, - emerge
dynamically, - interact with town economy and events.

SKILLS: Bellwether should have RPG-style skills/attributes such as: -
gardening, - cooking, - observation, - empathy, - persuasion, - physical
resilience, - practical skill, - occult knowledge, - and other suitable
abilities.

Skills should generally improve naturally through repeated relevant
activity rather than primarily through manual point allocation.

HEALTH AND NEEDS: Health is a major mechanic. Keep needs simpler than
The Sims and closer to RPG abstraction. Do not force the player to
micromanage every bodily need. Sleep, energy, food, illness, injury, and
psychological state may matter where they create gameplay value, but
avoid tedious maintenance.

PLAYER PSYCHOLOGY: Use creative freedom, with Disco Elysium as an
important reference. Possible presentation: - psychological
tendencies, - internal interpretations, - competing inner voices or
skill perspectives, - altered prose, - unreliable perceptions, -
hallucinations, - constrained or distorted choices, - vague state
descriptions, - selective explicit stats where useful.

Psychological systems should interact with: - town corruption, -
recurrence, - relationships, - player history, - anomalies, -
investigation, - LLM context, - and the town’s response to the player.

======================================================================
7. NPC POPULATION AND SOCIAL SIMULATION
======================================================================

Target population: - approximately 6–7 deeply simulated core NPCs, -
approximately 20 lighter background residents.

The current known core NPCs include: - Mara, - Jonah, - Mrs Ellis.

Future core NPCs should be designed according to story needs, missing
social roles, simulation roles, institutions, conflicts, and thematic
functions.

Background residents should preferably be named recurring people with
persistent identities, simple routines, relationships, and lightweight
memory. Some procedural generation may be used carefully, but once a
resident becomes established or interacted with meaningfully, identity
and state should persist.

NPCs should be autonomous during simulated game time. They may: - move
according to routines and goals, - meet, - talk, - exchange
information, - argue, - form opinions, - change relationships, - make
plans, - affect world state, - pursue personal goals, - progress stories
without the player witnessing every step.

Important NPCs should have goals independent of the player. Examples may
include: - wanting to leave, - protecting someone, - repairing a
relationship, - expanding a business, - hiding something, -
investigating something, - sabotaging someone, - changing career, -
responding to fear or corruption.

NPC-to-NPC relationships matter. NPCs can form friendships, conflicts,
romance, partnerships, breakups, marriage, jealousy, reconciliation, and
other relationships without the player being the centre of all social
life.

RELATIONSHIP MEMORY: Important NPCs should remember significant
interactions, including: - promises, - betrayals, - help, - gifts, -
shared experiences, - arguments, - important dialogue, - rumours, -
observed behaviour, - major choices.

Do not attempt to store every line forever. Memory should be selective,
meaningful, summarized, and available to constrained LLM context.

Two players at the same nominal relationship level may have very
different relationship histories.

KNOWLEDGE BOUNDARIES: This is extremely important: NPCs must know only
what they could plausibly know. An NPC cannot know a private fact unless
it reached them through: - direct conversation, - observation, -
evidence, - another NPC, - rumour propagation, - an institution, - or an
explicitly justified supernatural mechanism.

Information should propagate through the social network. Rumours may
mutate or be interpreted differently as they travel.

======================================================================
8. DIALOGUE AND THE LOCAL LLM
======================================================================

The local Ollama LLM is not intended to be a free-form omnipotent
storyteller. It is a constrained manager/orchestrator operating inside
the game engine.

The engine asks the LLM for suggestions about what the next state or
plausible developments should be. In this sense, the simulation is
partly managed by the LLM, but deterministic systems, authored truth,
content catalogues, state validation, and hard constraints remain
authoritative.

The LLM should help make Bellwether unpredictable and alive.

Desired LLM roles may include: - Director/world-round suggestions, -
bounded state transition proposals, - NPC decisions within available
legal options, - contextual dialogue wording, - free-form one-on-one
player/NPC dialogue, - NPC-to-NPC dialogue, - memory interpretation and
summarization, - rumour phrasing and propagation suggestions, -
environmental prose, - dreams, - psychological framing, - anomaly
presentation, - visual deception parameters, - event suggestions, -
playstyle inference, - contextual social situations and errands.

The LLM should progressively gain compact context about: - authored
story knowledge unlocked so far, - town state, - current
anomaly/corruption state, - player playstyle, - important player
history, - relationships, - significant memories, - recent events, - NPC
knowledge boundaries.

It must NOT: - invent canonical facts, - overwrite authored mystery
truth, - invent casual lore that becomes accidentally canonical, -
create impossible NPC knowledge without justification, - freely advance
story milestones, - hallucinate unavailable locations/items/people as if
canonical, - bypass deterministic validation, - destabilize save state.

The central story truth, major revelations, essential story beats, and
endings are authored and fixed. The route between authored anchors is
simulation-driven and unpredictable.

Example: The LLM may influence whether rain changes Jonah’s plans, Mara
hears a rumour, two residents quarrel, a social situation emerges, or a
psychologically appropriate anomaly is presented. It may not invent the
true explanation of Bellwether.

FREE DIALOGUE: One-on-one player/NPC conversation should support the
player typing whatever they want. The system should interpret free text
under: - NPC personality, - relationship history, - knowledge
boundaries, - memory, - current context, - emotional state, - world
truth constraints.

The current codebase already contains significant
LLM/Director/conversation foundations. Future work must inspect and
extend existing implementation rather than replace it.

======================================================================
9. STORY, MYSTERY, AND SUPERNATURAL MODEL
======================================================================

The central story is authored. Major story missions completed by the
player should NOT reset after failure/recurrence.

The story should support: - one true ending and/or multiple fixed major
authored endings, - meaningful authored revelations, - multiple
playstyle routes to knowledge, - continued life-sim/RPG play after
resolution.

Different playstyles should reveal different aspects of the mystery: -
social players: people, rumours, contradictions, relationships, -
explorers: geography, hidden areas, physical evidence, -
gardeners/farmers: ecology, seasons, growth anomalies, - workers:
institutions, economic patterns, records, routines, - investigators:
explicit evidence, deductions, archives.

The supernatural should eventually have a coherent explanation
accounting for approximately 95% of phenomena. The remaining 5% may
remain uncertain, contradictory, or genuinely unexplained.

The town is always fundamentally sinister/evil, even if recurrence or
player action sometimes produces positive outcomes.

Creative direction for the town’s nature: The town may behave as a
distributed psycho-geographic organism or intelligence. It need not
think exactly like a human. It may preserve patterns, feed on or
metabolize psychological states, reshape social pressure, manipulate
memory, alter probability, repeat motifs, and respond to how residents
collectively understand it. The final authored explanation should
determine the real mechanism; until then, systems should preserve
ambiguity and avoid random lore invention.

======================================================================
10. FAILURE, RECURRENCE, MEMORY, AND PERSISTENCE
======================================================================

Failure can come from multiple domains. Creative freedom is allowed.
Possible failure modes include: - physical death, - severe injury, -
psychological collapse, - social destruction/isolation, -
disappearance, - anomaly consumption, - dangerous investigation
outcomes, - being trapped, - other contextually justified states.

Failure should not be a simple game-over screen or total reset.

RECURRENCE/RECOVERY: After failure, the player may awaken: - at the
clinic, - at the graveyard, - at home, - somewhere impossible, - or in
another context determined by the town’s horror/anomaly state and the
nature of failure.

Story missions already completed remain completed.

Some possessions/inventory/progress may reset or be lost as part of
recovery, depending on system design and failure context.

THE TOWN LIVES ON AND REMEMBERS: Persistence should be a mixture of
literal and uncanny continuity.

Literal continuity may include: - NPC relationships continuing, - town
economy changing, - businesses or routines changing, - social
consequences remaining, - physical/world changes persisting, - residents
remembering events.

Uncanny continuity may include: - déjà vu, - dreams, - instinctive trust
or fear, - scars, - fragmented memories, - impossible knowledge, -
repeated phrases, - objects or plants surviving when they should not, -
altered attitudes without conscious explanation.

PLAYER MEMORY: The player character initially retains fragmented memory.
The longer the human player spends in the game and the more
recurrence/history accumulates, the more memory can return. Memory
recovery is a form of long-term progression.

HORROR ESCALATION AFTER FAILURE: Each failure/recovery can increase the
presence or accessibility of horror. This does not mean every recurrence
is simply numerically worse. Sometimes previous kindness or choices may
create positive echoes. However, the town remains fundamentally
sinister.

Escalation may move from: - coincidences, - wrong memories, - subtle
dialogue changes, - repeated phrases, - impossible familiarity, - small
map discrepancies, to: - altered geography, - missing residents, -
hostile manifestations, - corrupted routines, - dangerous rules, -
physical threats, - interface deception, - major psychological
distortion.

======================================================================
11. HORROR PHILOSOPHY
======================================================================

Primary horror modes: - psychological, - social, - relational, -
epistemic, - environmental, - systemic.

Physical danger exists but is less dominant and should be used
selectively.

Horror should be stronger because the player first learns and loves
ordinary Bellwether.

Safe routines, people, places, objects, and rules may exist. Creative
freedom is encouraged in designing them. Some may later fail, mutate, or
reveal conditions the player misunderstood.

Potential town effects include: - social atmosphere shifts, - altered
weather patterns, - changed paths, - changed buildings, - collective
dreams, - altered or conflicting memories, - aggression, -
compulsions, - disappearances, - repeated events, - impossible
geography, - economic anomalies, - illness, - corrupted routines, -
ecological abnormalities, - temporal inconsistencies.

These effects must be grounded in authored state and constrained
systems. The LLM may choose or parameterize allowed manifestations but
should not invent unbounded canon.

======================================================================
12. INTERFACE AND VISUAL HORROR
======================================================================

The UI may become unreliable as part of horror.

Creative possibilities include: - map geography changing, - discovered
routes disappearing, - impossible routes appearing, - portraits subtly
changing, - text correcting itself, - journal entries changing, - false
notifications, - duplicated dialogue choices, - altered quest wording, -
impossible timestamps, - save-slot anomalies, - brief scenes the game
later denies, - inconsistent labels, - subtle visual corruption.

Hard rule: The game must remain playable unless a legitimate failure
mode has been triggered.

Interface horror must not become random usability sabotage. It should
be: - state-driven, - meaningful, - bounded, - recoverable where
appropriate, - readable enough for the player to continue.

The ordinary UI should remain clean and uncluttered.

======================================================================
13. MAP, GEOGRAPHY, TRAVEL, AND EXPLORATION
======================================================================

Bellwether has an illustrated canonical map and a dynamic exploration
map system.

Map principles: - only explored geography should be visible, -
exploration persists, - player location is shown with a “You are here”
marker, - the legend/key and compass should remain permanently visible
and must not reveal hidden geography, - fog should resemble blank
textured parchment, - explored areas should merge into a continuous
feathered reveal, - routes and regions should reveal naturally rather
than as hard circular spotlights or disconnected cut-outs, - map state
should derive from authoritative world geography and player knowledge,
not duplicate the world model.

Current near-term world scope: - village, - immediate surroundings, -
woods, - quarry, - farms, - caves.

Future scope may expand to: - a larger town, - more regions, -
neighbouring settlements, - transport links, - additional exploration
zones.

TRAVEL: Travel design should suit the game rather than become
unnecessary friction. Recommended direction: - ordinary travel costs
in-game time, - important walks can produce short observations, -
weather can affect travel, - exploration can reveal routes and
shortcuts, - encounters may happen, - social meetings may occur, -
discoveries may be made, - danger appears selectively as corruption
rises.

Do not force long repetitive prose on every routine trip. Use a layered
system: - quick travel-like resolution for familiar safe routes when
appropriate, - richer eventful travel when conditions, novelty,
encounters, weather, or horror justify it.

======================================================================
14. QUESTS, PROCEDURAL CONTENT, AND PLAYER DIRECTION
======================================================================

Bellwether should support: - authored main story quests, - authored side
quests, - visible quest log, - journal/notebook, - procedural
LLM-suggested errands, - procedural social situations, - dynamic
opportunities from economy/jobs/relationships.

Procedural content must coexist with authored content without confusing
the two.

Procedural errands should be grounded in: - real NPC needs, - existing
items, - actual locations, - current relationships, - jobs, - economy, -
weather, - schedules, - current world state.

The LLM must not fabricate arbitrary quest canon.

The player may ignore the main story, but the game can surface reminders
through: - rumours, - consequences, - revelations, - corruption, -
pressure, - NPC concern, - changing conditions.

======================================================================
15. CHARACTER PRESENTATION
======================================================================

Core NPCs should have small left-side UI portraits.

Portrait direction: - small identity portrait, -
head-and-shoulders/passport-photo composition, - soft-painted rural
anime visual novel style, - restrained semi-realistic anime
proportions, - expressive but natural facial anatomy, - painterly
texture, - warm contemporary rural English atmosphere, - subtle natural
lighting, - muted pastoral colours, - detailed eyes without exaggerated
anime styling, - face readable at small UI size, - simple unobtrusive
background, - no scenery, - no text, - no border, - no dramatic pose, -
no fantasy styling.

For now, portraits should remain FIXED. Do not create large
expression-variant asset sets yet. This avoids asset bloat.

Known portrait needs: - Mara, - Jonah, - Mrs Ellis, - future core NPCs
as they are designed.

Mrs Ellis must look genuinely older, not like a young anime character
with grey hair.

======================================================================
16. BACKGROUND ART AND SCENE PRESENTATION
======================================================================

Major locations should eventually have illustrated background scenes.

Visual direction: - soft-painted anime environment art, - detailed
painterly VN background, - contemporary rural English village, -
restrained semi-realistic anime aesthetic, - muted pastoral colours, -
natural lighting, - lived-in environmental storytelling, - no visible
characters unless intentionally authored.

Each location should have one canonical composition first. Do not
independently regenerate every weather/season/time version from text,
because architecture and camera geometry will drift.

Variant workflow: - canonical approved scene, - reference-image edit for
rain, - reference-image edit for overcast, - reference-image edit for
autumn, - reference-image edit for winter, - reference-image edit for
snow, - reference-image edit for evening/night.

Preserve: - camera position, - perspective, - architecture, - path
geometry, - fences, - major objects, - skyline, - landmark positions.

Do not create a full Cartesian product of season × weather × time for
every location. Use selected authored variants for important locations
and lighter overlays/effects where suitable.

======================================================================
17. AUDIO DIRECTION
======================================================================

Bellwether needs audio.

Desired layers: - comforting ordinary village ambience, - location
themes, - seasonal themes, - weather ambience, - interior/exterior
ambience, - subtle social spaces, - horror corruption of familiar audio.

As corruption rises, familiar audio may: - detune, - degrade, - reverse
selectively, - acquire impossible sounds, - lose expected sounds, -
introduce distant voices or motifs, - mismatch location or weather, -
stop entirely at meaningful moments.

Audio horror should be state-driven, not random noise.

======================================================================
18. PLAYER IDENTITY AND CHARACTER FORMATION
======================================================================

At the start, character creation should ask for NAME ONLY.

The protagonist has some authored backstory, but the player character
should develop into a virtual person through play.

Over time, the game builds a model of the player character from: -
choices, - activities, - work, - relationships, - dialogue, - habits, -
investigation style, - generosity/self-interest, - risk tolerance, -
routines, - psychological responses.

NPCs and the town form an image of who the player is.

The human player’s behaviour should gradually create: - reputation, -
personality tendencies, - playstyle model, - relationship identity, -
psychological profile, - social expectations.

The player character is therefore neither a fully blank avatar nor a
rigid authored protagonist.

======================================================================
19. REPLAYABILITY AND WORLD VARIATION
======================================================================

A new game should be able to produce a meaningfully different Bellwether
while preserving authored mystery truth.

Variable elements may include: - initial weather and weather history, -
social conflicts, - NPC relationship trajectories, - job
opportunities, - economy, - rumours, - event timing, - encounters, -
anomaly patterns, - environmental conditions, - procedural errands, -
social situations.

The current project already uses LLM/Director influence for some initial
and evolving conditions such as weather. Preserve and extend existing
architecture.

Replayability must come from systemic variation around authored truth,
not from randomly changing the actual explanation of the mystery.

======================================================================
20. PACING AND LENGTH
======================================================================

No rigid target playtime is currently mandated.

Recommended direction: - allow open-ended life-sim play, - let authored
story progression depend partly on player engagement, - support a
substantial first playthrough rather than rushing the mystery, - ensure
meaningful ordinary-life time before major horror escalation, - allow
different players to reach endings at different real-world playtimes.

A likely healthy target can be determined through exhaustive
player-perspective playtesting rather than fixed prematurely.

Pacing goals: - early attachment and curiosity, - rich ordinary
routine, - gradual anomalies, - escalating interpretive uncertainty, -
increasing pressure, - major revelations, - authored ending routes, -
continued post-story life-sim play.

======================================================================
21. PLAYTESTING STANDARD
======================================================================

Bellwether releases must be tested from the player’s perspective.

Do not rely only on: - unit tests, - syntax checks, - deterministic
ending simulations, - diagnostics.

Also perform: - full playthrough simulation, - different choice
patterns, - different playstyles, - reading prose and UI options in
sequence, - checking whether story is understandable without code
knowledge, - pacing evaluation, - run-length evaluation, - ordinary-life
activity before horror escalation, - meaningful player freedom, -
save/load testing, - recurrence testing, - old-save compatibility where
applicable, - LLM-off/fallback behaviour where applicable, - UI
readability, - absence of soft-locks, - verification that existing
features still work.

The user wants to remain relatively unspoiled and discover the proper
story through play. Development playtesting should protect that goal
where possible.

23. RECOMMENDED HIGH-LEVEL DEVELOPMENT DIRECTION
======================================================================

Near-term emphasis should be on making the existing game feel alive and
enjoyable moment-to-moment, not endlessly adding invisible
infrastructure.

Recommended broad priorities:

A. COMPLETE ORDINARY-LIFE VERTICAL SLICE A player should be able to: -
wake at the cottage, - choose morning activities, - manage simple
health/energy/food, - garden or cook, - travel through the village, -
encounter a portraited NPC, - have contextual free-form conversation, -

C. DAILY LIFE DEPTH - gardening, - cooking, - jobs, - economy, - cottage

D. SPATIAL LIFE - meaningful travel, - map discovery, - encounters, -

E. NPC PURPOSE AND SOCIAL LIFE - schedules, - goals, - exceptions, -

F. AUDIO - ambience and music, - later corruption of familiar sound.

G. HORROR THROUGH CORRUPTION OF FAMILIARITY - do not rush horror before

======================================================================
24. CORE DESIGN TEST FOR EVERY NEW FEATURE
======================================================================

Before adding a feature, ask:

1.  Does it make ordinary Bellwether more enjoyable, alive, or
    meaningful?
2.  Does it create persistent consequences?
3.  Does it interact with existing systems rather than sit isolated?
4.  Can NPCs or the town respond to it?
5.  Can it later support horror through corruption of something
    familiar?
6.  Does it preserve player autonomy?
7.  Does it avoid unnecessary UI clutter?
8.  Does it preserve existing working features?
9.  Does it keep authored truth separate from LLM improvisation?
10. Will the player be excited to return and see what happens next?

If a feature fails most of these tests, reconsider it.

======================================================================
25. ONE-SENTENCE VISION
======================================================================

Bellwether is an open-ended rural village life-sim, RPG, interactive
fiction, and psychological/social horror game in which the player builds
a real daily life inside a deeply simulated, LLM-managed but authorially
constrained community, while a sinister living town remembers, changes,
corrupts familiar reality, and gradually reveals a mostly explainable
mystery across persistent relationships, failures, recurrence, and
player-shaped identity.


======================================================================
26. ADDITIONAL CANONICAL DIRECTION: CONFINEMENT, PROTAGONIST, ROMANCE,
CAST CONSEQUENCES, DANGER, AND ENDGAME
======================================================================

CONFINEMENT:
Leaving Bellwether is effectively impossible, but the mechanism is primarily psycho-geographic rather than a simple visible wall or conventional locked boundary.

The town affects the psyche. Attempts to leave may be undermined by:
- compulsions to return,
- dissociation,
- memory gaps,
- rationalization,
- panic or physical distress,
- routes that somehow return to familiar places,
- abandoned intentions,
- altered perception of distance or destination,
- social pressure and dependency,
- selective inability to sustain departure.

This should initially be deniable. Residents may have different explanations for why they remain. Some may insist they simply chose to stay. Others may have tried repeatedly to leave. The authored mystery should eventually explain the confinement mechanism as part of the broader nature of Bellwether.

PROTAGONIST BACKSTORY DIRECTION:
Recommended authored foundation:

The protagonist comes to Ashcroft Cottage after inheriting or receiving legal possession of it through a family connection that is real but poorly understood. The connection should not make the protagonist a preordained chosen one. Instead, it provides a plausible reason for arrival and a thread into the broader mystery.

The protagonist's family history may contain:
- a relative who lived in or passed through Bellwether,
- incomplete correspondence or records,
- contradictory memories,
- an unexplained gap in family history,
- a prior attempt to leave Bellwether,
- a connection to an earlier recurrence or containment event.

The protagonist arrives partly seeking a quieter life and partly because the cottage offers an affordable practical restart. Their deeper connection to Bellwether should emerge gradually through fragmented memory and investigation.

The exact secret history must be authored in the Story Bible and should not be improvised by the LLM.

ROMANCE AND FAMILY LIFE:
Romance may progress beyond dating into:
- long-term partnership,
- cohabitation,
- marriage,
- and potentially children.

These systems should be developed carefully and late enough that they integrate with the persistent simulation, economy, housing, NPC autonomy, recurrence, and post-story life-sim.

Relationships should remain dynamic after commitment. Marriage is not an endpoint that freezes an NPC's personality or independent goals.

CORE NPC CONSEQUENCE POLICY:
Recommended rule: deep autonomy with protected authored gates.

Core NPCs may:
- become hostile,
- distance themselves,
- form relationships with others,
- marry others,
- change jobs,
- suffer injury,
- disappear temporarily,
- leave ordinary routines,
- make major independent decisions.

However, permanent death, permanent disappearance, or irreversible departure of a core NPC should require an authored or validated major-event gate. The LLM may suggest circumstances and timing within constraints, but it cannot casually delete a core character or make essential story content impossible.

Lighter residents may have more permissive systemic outcomes, provided continuity and plausibility are preserved.

DANGER AND CONFLICT:
Bellwether should NOT become a conventional combat RPG.

Danger should be resolved primarily through:
- choices,
- skills,
- items,
- preparation,
- relationships and help,
- environmental knowledge,
- escape,
- hiding,
- timing,
- health checks,
- psychological resilience,
- prior investigation.

Physical confrontations may occur, but the primary gameplay is decision-driven and systemic rather than based on a dedicated attack/defence combat loop.

ENDGAME EMOTIONAL SHAPE:
The supernatural threat can be neutralized or contained to a meaningful extent, but Bellwether does not need to become an entirely ordinary place.

A successful resolution may:
- stop the worst predatory or escalating behaviour,
- break or weaken recurrence mechanisms,
- restore greater agency to residents,
- reduce psychological coercion,
- establish boundaries or rules,
- transform the relationship between town and residents,
- allow informed coexistence.

The player may ultimately understand enough to live in Bellwether knowingly rather than merely remain trapped unknowingly.

The strongest ending direction is neither total destruction of the supernatural nor helpless surrender. It is costly understanding, partial neutralization, restored agency, and coexistence with something that remains strange.

BELLWETHER — CANONICAL WORLD AND STORY BIBLE Version 1.0 |
Developer-facing spoiler document | 2026-07-08

PURPOSE This defines Bellwether’s canonical world, hidden supernatural
truth, story architecture, historical layers, mystery threads, cast
direction, recurrence, endings, and canon boundaries. It is a spoiler
document. Do not expose it wholesale to players or local LLM prompts.
Use it with the Comprehensive Project Context v2 and Engineering Context
Snapshot v0.3.4. Existing working code remains authoritative for
implementation; integrate this canon additively and preserve existing
systems.

====================================================================== 1.
CORE CANON
======================================================================

Bellwether is a tiny contemporary rural English settlement, somewhat
larger than the current playable map but socially dense and small enough
for repeated faces, institutions, family histories, and rumours to
matter.

Bellwether was not built and then haunted. Human settlement grew inside
the territory of an ancient supernatural distributed intelligence. Over
centuries, habitation gave it structures through which to express
itself: roads became pathways, institutions became organs, routines
became rhythms, relationships became connections, memories became
storage, and traditions became regulatory mechanisms.

The village and entity are deeply entangled but not identical. Human
affection, work, conflict, love, and history are genuine. Residents are
not puppets.

The entity is supernatural—not a simulation, experiment, alien
technology, collective hallucination, or conventional demon. Its
consciousness is non-human and distributed. A hive mind is a useful
analogy, but it is better understood as a psycho-geographic nervous
system expressed through place, memory, repetition, social networks,
dreams, probability, and living minds.

The entity predates the player. Eleanor Ashcroft’s final intervention
left it relatively dormant. The first game’s opening begins a new
wakefulness phase. Ordinary residents do not start with a shared secret
knowledge of the supernatural; they experience the awakening, form
opinions, disagree, rationalize, investigate, and change.

Central theme: Can a place give you home, love, purpose, and belonging
without acquiring ownership of you? Bellwether genuinely offers
belonging. Its evil lies partly in its inability to distinguish
belonging from possession.

======================================================================
2. THE CHORUS — DEVELOPER TRUTH
======================================================================

Working developer designation: THE CHORUS. This need not be its true
name or common in-world terminology.

The Chorus is an ancient supernatural intelligence localized beneath and
around Bellwether. It has no single body, throne, heart, or brain that
can simply be killed.

It expresses itself through human memory, social imitation, dreams,
repeated routines, emotional contagion, geography, paths and thresholds,
bells and rhythmic sound, architecture, burial sites, watercourses,
probability, collective expectation, recurrence, and occasional
biological/environmental correlates. Fungal or plant networks may
correlate with manifestations but are not the scientific cause of the
supernatural.

The Chorus does not read every mind continuously. Awareness is strongest
where attention, repetition, emotion, and social connection overlap. It
learns a person through routines, fears, attachments, choices,
remembered places, social roles, self-concept, and other people’s
beliefs about them. Therefore the game’s player-style model, habits,
relationships, skills, and memories have narrative significance: they
are also the shape through which the town learns the player.

LIMITS: - It cannot freely rewrite all reality. - It cannot know facts
no connected mind/place has carried. - It cannot create arbitrary matter
without cost or pattern. - It cannot instantly control healthy minds. -
It cannot sustain unlimited physical manifestations. - It cannot
perfectly erase deeply reinforced memories. - It struggles against
independently maintained, mutually reinforcing anchors. - It cannot
fully dominate people whose identity and relationships remain resilient
and mutually reinforced.

These limits are essential to investigation and the true ending.

======================================================================
3. DRIVES
======================================================================

The Chorus has overlapping non-human drives: CONTINUITY: resists
abandonment, depopulation, rupture, and forgotten patterns. COHERENCE:
pushes toward shared interpretations, synchronized routines, repeated
routes, and reinforced roles. INCORPORATION: learns newcomers and fits
them into the social/mnemonic network. RETENTION: experiences departure
by deeply incorporated people as loss or mutilation. STIMULATION: strong
emotion makes the network more active and legible to itself; fear is
efficient but not the only stimulus. CORRECTION: tries to restore
learned patterns when people/events diverge. SELF-PRESERVATION: can
isolate, deceive, divide, injure, or kill when sufficiently threatened.

These drives may conflict. The town can save someone physically because
it wants to retain them and then psychologically harm them while forcing
them into a stable role. The entity is sinister without every event
being simple sadism.

======================================================================
4. CONFINEMENT AND OUTSIDE CONTACT
======================================================================

Entry is easier than exit. At first Bellwether still receives post,
deliveries, intermittent buses, phone/internet, outside news, goods,
visitors, and occasional newcomers. Information and goods cross more
easily than incorporated people.

LOW WAKEFULNESS: postponed plans, anxiety near departure,
rationalization, missed tickets, forgotten bags, sudden obligations,
confusing roads, unexplained returns. MIDDLE: looping roads, lost time,
GPS/sign disagreement, buses returning unexpectedly, memories of
reaching places contradicted by evidence, inconsistent passenger
accounts. HIGH: overt impossible geography, self-returning routes,
disappearance between landmarks, severe psychological/physical distress,
direct manifestations or dangerous events.

Different people experience the boundary differently according to
incorporation depth, identity stability, social anchors, recurrence
sensitivity, and town state.

======================================================================
5. PROTAGONIST, ELEANOR, AND ASHCROFT COTTAGE
======================================================================

The protagonist comes to Bellwether after inheriting or receiving legal
possession of Ashcroft Cottage through Eleanor Ashcroft, a distant but
genuine maternal relative such as a great-aunt. The player knew of her
only vaguely.

The practical motive is ordinary: an unexpected property and a chance
for a restart. The protagonist may initially intend to inspect, repair,
live in, sell, or decide later. They are NOT a chosen one.

A branch of the family once partially escaped Bellwether. Weak residual
effects persisted through family story, behaviour, objects, and possibly
supernatural imprint: dreams, missing stories, aversion to certain
sounds/routes, inexplicable familiarity, repeated phrases.

ELEANOR: Publicly eccentric, capable, private, interested in local
history and gardening, increasingly withdrawn. Privately she
investigated failed departures, contradictory records, memories, and
routes for decades.

Her key discovery was ANCHORING: isolated memories and records are easy
to correct; independent, mutually reinforcing relationships, records,
routines, and meaningful physical objects resist correction.

She created a distributed archive across letters, gardening records,
recipes, marginalia, repair notes, church and clinic records, council
maps, receipts, photographs, testimony, hidden objects, route
measurements, and recurring phrases. She hid truth inside ordinary life
because centralized archives were vulnerable to alteration,
disappearance, human suppression, or social discrediting.

Canonical truth: Eleanor attempted a partial containment that reduced
the Chorus to dormancy but could not free Bellwether. It cost her life
or displaced her beyond ordinary recoverability. Legal death and
physical fate may not perfectly align. She did not summon the player as
sacrifice; she created a possibility for a successor.

ASHCROFT COTTAGE: Home first, memory anchor second, mystery site third.
It must remain useful, restful, upgradeable, customizable, and central
to gardening, cooking, storage, relationships, and ordinary life.

Eleanor reinforced it through records, modifications, garden cycles,
cross-referenced journals, meaningful objects, thresholds, and routines.
It resists large corrections better than many places but is not
perfectly safe. Restoration can uncover distributed evidence. Gardening
can reveal temporal/ecological inconsistency. Cooking and recipes may
carry memory without turning every recipe into a puzzle.

======================================================================
6. RECURRENCE
======================================================================

Recurrence is the Chorus attempting to preserve coherence under
catastrophic contradiction. Historically there were subtle localized
corrections: lost days, repeated decisions, incompatible memories,
partial repetitions.

The player’s recurrence becomes unusually visible because of inherited
residual imprint, Eleanor’s anchors, dense relationships, investigation,
repeated exposure, awakening, and an identity increasingly legible to
the town.

Completed authored story missions do not reset after failure.

Recovery may occur at clinic, cottage, graveyard, roadside, an
institution, or somewhere contextually impossible. Location depends on
failure type, anomaly state, relationships, discovered anchors, story
phase, and wakefulness.

Continuity is literal and uncanny. Literal persistence may include story
milestones, social consequences, relationship changes,
economy/institutional changes, repairs, discoveries, and canonically
persistent deaths. Uncanny persistence includes dreams, instinctive
trust/fear, scars, repeated phrases, unexplained familiarity, altered
objects, impossible plant survival, and emotional memory without factual
memory.

Player memory returns gradually through time, recurrence, investigation,
anchors, relationships, and skills.

NPC sensitivities differ: emotional recognition, procedural memory,
bodily route panic, factual fragments without source, documentary
recognition, or strong resistance. Romantic partners can be powerful
anchors, but equally valid non-romantic anchors must exist.

======================================================================
7. POPULATION AND CORE CAST
======================================================================

Target: 6–7 deeply simulated core NPCs, about 20 lighter named recurring
residents, plus implied population. Children may exist naturally in
population/family simulation; they are not a special prohibition on
darkness, but must never be used merely as cheap shock devices.

CORE FUNCTIONS: MARA — emotional/social anchor; land, garden, ecology,
practical knowledge. She discovers truth through noticing biological,
seasonal, soil, weather, and animal inconsistencies. Recurrence
sensitivity is emotional/sensory familiarity.

JONAH — social warmth, bakery, food economy, gossip crossroads, routine.
He notices repeated orders, phrases, missing customers, impossible
familiarity, and community rhythm. Recurrence sensitivity is procedural
memory.

MRS ELLIS — older resident, social memory, observation, bridge between
gossip and historical contradiction. Not a secret mastermind. She
remembers incompatible versions of events. Sensitivity: factual
fragments with unreliable chronology.

CLINIC FIGURE (Dr Vale or equivalent) — health, treatment, injury,
psychological observation, records. Starts rational/skeptical and is
forced toward theory by repeated clinical patterns. Sensitivity: pattern
recognition in injuries, scars, physiological responses, treatment
histories.

COUNCIL/ARCHIVE FIGURE — property, maps, planning, administration,
records, jobs. Exposes map, census, ownership, road, and bureaucratic
contradictions. May become morally ambiguous. Sensitivity: documentary
recognition.

YOUNGER RESIDENT DETERMINED TO LEAVE — tests the boundary and gives
confinement emotional immediacy. Attempts create evidence. Can be
friend, rival, romance, or tragedy route. Sensitivity: bodily panic and
route memory.

RECENT ARRIVAL / SOCIAL OUTSIDER — comparative perspective, challenges
normalization, demonstrates entry is possible, provides baseline for
incorporation. May become ally, skeptic, opportunist, or antagonist.

Exact names, ages, designs, and quest chains for new cast remain
reserved for implementation.

Core NPCs have strong autonomy. Permanent death, permanent
disappearance, or irreversible departure requires authored/validated
major-event gates. LLMs cannot casually remove core cast.

======================================================================
8. HUMAN RESPONSES AND INSTITUTIONS
======================================================================

There is no conventional singular villain and no standard hooded cult.
Human antagonism comes from incompatible partial truths.

Informal positions may emerge: KEEPERS — old customs and silence
preserve stability. BREAKERS — disruption and forced escape are worth
severe risk. ACCOMMODATIONISTS — coexistence is possible, sometimes with
morally compromised choices. EXPLOITERS — rare people attempt to use
anomalies, prediction, memory effects, or influence.

No faction knows everything and important NPCs should not all begin as
secret faction members.

INSTITUTIONS: CLINIC: repeated injuries, scars, stress responses,
recurrence traces. COUNCIL: maps, boundaries, roads, missing addresses,
censuses, planning contradictions. CHURCH: not proof of demons;
preserves centuries of misinterpreted observations, bell schedules,
burial placement, processions, stories of wandering. BUSINESSES:
repeated orders, stock discrepancies, deliveries, missing customers,
rumours, ordinary rhythm. FARMS/GARDENS: seasons, growth timing, seed
continuity, animals, soil. GRAVEYARD: names, dates, identity
contradictions, recurrence associations, possible recovery site.

======================================================================
9. CUSTOMS
======================================================================

No conventional cult. Ordinary customs evolved from forgotten regulation
attempts: - bell ringing at specific intervals; - gates left unlatched
on certain nights; - communal meals; - annual routes walked in fixed
directions; - boundary hedges; - old stones never moved; - personal
objects buried with the dead; - lights in certain windows; - seasonal
food exchange; - avoidance of counting repeated sounds aloud.

Residents understand these as tradition, superstition, neighbourliness,
or habit. Some customs help, some are obsolete, some misunderstood, and
some strengthen the Chorus. The player must learn which through
evidence.

======================================================================
10. HISTORICAL LAYERS
======================================================================

PRE-SETTLEMENT: territory avoided, used seasonally, or marked by lost
protective practice. MEDIEVAL: permanent settlement strengthens
structure; records of lost travellers, impossible bells, returned dead,
collective dreams, processions restoring order. Contemporary people
interpret events religiously, but the cause is not demonic. EARLY
MODERN: customs become traditions and knowledge fragments across
families/institutions. VICTORIAN: doctor/naturalist/surveyor circle
studies routes, synchronization, memory contradiction, sound and
routine. They fail partly because they centralize evidence; records
vanish, alter, or are discredited. 20TH CENTURY: a major institutional
incident involving infrastructure, evacuation, wartime use, medicine, or
road modernization creates cover-up, missing records, and contradictory
casualty/disappearance counts. Exact incident reserved for later
authoring. ELEANOR’S DECADES: distributed investigation and anchoring
discovery. ELEANOR’S CONTAINMENT: low-wakefulness interval. PRESENT:
player arrives; awakening resumes.

======================================================================
11. MYSTERY THREADS
======================================================================

Several apparently separate mysteries gradually reveal one system.

THE ROADS — failed departures, route logs, buses, maps, GPS
discrepancies, missing time. THE WRONG MEMORIES — incompatible
testimony, photographs, diaries, relationships, recurrence echoes.
ELEANOR — what she investigated and what happened; cottage, garden,
letters, distributed archive. THE BELL AND CUSTOMS — why traditions
affect anomalies; church records, timing, oral history, experiments. THE
LIVING LAND — ecology/weather/routes/mood correlations; gardening,
farming, animals, seasons. THE BODY REMEMBERS — injuries, scars, panic,
habits; clinic, health, recurrence. THE SOCIAL CHORUS — rumours,
repeated phrases, mood synchronization, roles; NPC simulation and
knowledge propagation. THE PLAYER — family history, Eleanor, recurrence,
identity, relationship anchors.

Threads can be discovered in different orders.

======================================================================
12. REVELATION LADDER
======================================================================

0 HOME — ordinary, comfortable, alive. 1 COINCIDENCE — repetition,
familiarity, social echoes, weather correlations. 2 CONTRADICTION —
records, memories, routes, testimony disagree. 3 PATTERN — anomalies can
be proven non-isolated. 4 RESPONSIVENESS — town reacts to attention,
routine, relationship, disruption. 5 DISTRIBUTION — no single haunted
object/location; phenomenon operates through networks. 6 INCORPORATION —
town learns people; inability to leave links to incorporation. 7
ELEANOR’S METHOD — distributed anchors and ordinary life resist
correction. 8 RECURRENCE — failure/return understood as correction and
preservation. 9 CHORUS MODEL — distributed supernatural intelligence
using village structures. 10 RELATIONSHIP CHOICE — final problem is
stopping possession while preserving community and agency.

No NPC explains all stages.

======================================================================
13. HORROR RULES
======================================================================

Primary horror: psychological, social, relational, environmental,
epistemic, systemic. Physical horror is rarer.

Anomaly families: memory contradiction, synchronization, repeated
language, impossible familiarity, shared dreams, route distortion,
temporal discrepancy, record alteration, environmental mismatch, sound
anomalies, identity confusion, selective absence, recurrence echoes,
rare physical manifestations.

Body horror may exist but must be rare and biologically grounded in
presentation: plausible infection-like changes, abnormal wound healing,
scar patterns, coherent tissue response, realistic injury, stress
physiology, sleep deprivation, biologically plausible plant/fungal
interaction. Avoid constant impossible flesh spectacle. Physical
manifestations should feel costly and exceptional.

Interface deception may be used but the game remains playable unless a
legitimate failure mode occurs.

======================================================================
14. BELLWETHER MOTIF
======================================================================

The title is canonically meaningful. A bellwether leads a flock and
carries the bell others follow.

Motif layers: - church bell synchronizes communal time; - social leaders
transmit mood; - rumours propagate through influential residents; - the
player may become a bellwether; - the Chorus can use individuals to
entrain groups; - the true ending breaks compulsory synchronization
without destroying community.

Do not over-explain the metaphor in dialogue.

======================================================================
15. TRUE ENDING AND ENDINGS
======================================================================

The deepest resolution should be difficult, long, and often require
recurrence. It cannot be reached by collecting plot coupons alone.

Potential requirements: broad investigation, multiple mystery threads,
stable player identity, meaningful cross-role relationships, at least
one strong mutual anchor, geographic understanding, recurrence
experience, custom knowledge, ordinary-life participation sufficient to
know normality, institutional access, preparation/health, and key
Eleanor discoveries.

ENDING FAMILIES: INCORPORATION — peaceful-looking stable role at the
cost of agency. FALSE ESCAPE — apparent escape later shown incomplete;
use sparingly. RUPTURE — network destabilized without sufficient
anchors; Chorus injured but community suffers. ACCOMMODATION — limited
stable coexistence without general freedom. CONTAINMENT — improved
version of Eleanor’s dormancy; threat suppressed, relationship
unresolved. TRUE LIBERATION-COEXISTENCE — distributed anchors, social
understanding, institutions, ordinary-life networks, and recurrence
knowledge break compulsory ownership. Roads work, residents can leave,
coercion weakens, recurrence breaks or radically diminishes, town
remains supernatural, some residents leave and some return voluntarily,
player can continue living there.

True-ending emotional tone: relief, melancholy, earned affection, grief,
lingering unease. The player has not made Bellwether ordinary; they have
changed belonging from captivity toward consent.

======================================================================
16. POSTGAME, ROMANCE, AND CONSEQUENCE
======================================================================

After resolution the game remains playable. Postgame supports farming
expansion, jobs, economy, cottage development, hobbies, relationships,
festivals, side mysteries, lower-intensity anomalies, visitors,
departures, and returns.

Romance is optional. It may progress through attraction, dating,
conflict, reconciliation, cohabitation, marriage, and children where
appropriate. NPC autonomy continues after commitment. Romantic partners
can be strong recurrence anchors, but deep friendship, community trust,
family-like bonds, and institutional networks must provide non-romantic
alternatives.

Authored death routes are allowed. Core deaths require
authored/validated gates, causal legibility, and meaningful prevention
or consequence. The LLM cannot decide them casually. Recurrence does not
automatically trivialize death; story canon determines which
consequences persist literally or contradictorily.

======================================================================
17. STORY DELIVERY
======================================================================

The player infers truth through records, testimony, contradictions,
geography, recurrence, institutional evidence, ecology, social
simulation, dreams, experiments, and consequences.

Avoid final villain monologues, omniscient lore dumps, NPC omniscience,
random cryptic writing without eventual meaning, and arbitrary twists.

The ending confirms understanding through what works and what changes.

About 95% should be coherent and inferable. Deliberate residual
uncertainty may concern the Chorus’s ultimate origin, whether similar
entities exist, whether certain late anomalies are residual or
independent, the metaphysics of inherited imprint, and Eleanor’s final
subjective moment.

======================================================================
18. LLM CANON BOUNDARIES
======================================================================

NEVER insert this Bible wholesale into routine LLM context.

PUBLIC TIER: visible locations, weather/season, known NPC identity,

distributed nature of the entity.

The LLM improvises expression and bounded circumstance. It does not
improvise canon.

22. CANONICAL SUMMARY
======================================================================

Bellwether is a small contemporary English settlement built within the
territory of an ancient supernatural distributed intelligence. Over
centuries, roads, institutions, routines, relationships, memories,
customs, and bells became structures through which the entity—the
Chorus—perceives and preserves itself. It gives residents genuine
community while progressively incorporating them into a network that
cannot tolerate departure. Eleanor Ashcroft discovered that independent
relationships, ordinary routines, physical records, and mutually
reinforcing memories can resist the town’s corrections. Her final
containment left Bellwether dormant and created conditions for a
successor. The protagonist inherits Ashcroft Cottage and arrives for
ordinary reasons, beginning a new awakening. As the player builds a
life, the town learns them; the same bonds can become instruments of
possession or anchors of resistance. Across village life, investigation,
social change, failure, recurrence, and fragmented memory, several
mysteries resolve into one system. The deepest resolution is not
destroying community or killing a monster, but breaking compulsory
incorporation: restoring the possibility of departure, weakening
recurrence and coercion, and allowing Bellwether to remain strange
without owning everyone who belongs there.

BELLWETHER — LLM ROLE, AUTHORITY, AND CONSTRAINT SPECIFICATION Version
1.0 | 2026-07-08 | Authoritative AI-system design document

PURPOSE This document defines what LLMs do in Bellwether, what they may
influence, what they may never control, how models are routed, how
memory and scheduling work, and how hallucination is prevented. Use with
the Comprehensive Project Context v2, Engineering Context Snapshot
v0.3.4, and Canonical World and Story Bible v1.

CORE PRINCIPLE The deterministic engine is Bellwether’s physics, law,
authoritative memory store, and final state authority. The LLM system is
the village’s brain, ecology manager, social director, environment
driver, and adaptive storyteller. Developers define vocabulary, laws,
canon, primitives, and validators. LLMs compose novel developments from
legal elements. Bellwether should be emergent without becoming
incoherent.

====================================================================== 1.
CENTRAL GOAL
======================================================================

Two players reaching Day 30 after different choices should be able to
inhabit substantially different Bellwethers. LLM-driven differences may
include jobs changing, friendships forming or collapsing, romances
changing, businesses struggling or thriving, ambitions changing,
routines evolving, local conflicts, reconciliations, visitors, social
prominence of lightweight residents, ecological patterns, animals,
vehicles, weather character, seasonal abnormalities, rumours, NPC
theories, procedural opportunities, multi-day arcs, and adaptive horror.

All persistent changes must be legal, causal, validated, auditable, and
stored in authoritative game state. Bellwether must not merely contain
AI dialogue; the village itself should behave as an emergent phenomenon.

======================================================================
2. CURRENT IMPLEMENTATION BASELINE
======================================================================

The inspected v0.3.4 code already provides a strong foundation: - Ollama
integration and local CPU operation; - default qwen3:1.7b
configuration; - configurable model, URL, timeout, retries, and
threads; - bounded choices; - free-form player/NPC conversation; -
weather, NPC, traffic, and conversation Directors; - compact context
routing; - legal candidate generation; - parsing/repair; - deterministic
fallback; - AI health information and debug traces; - foreground
dialogue priority; - serialized calls for low-end hardware; - compact
story/player/session/run summaries; - recurrence-aware compressed
history.

Preserve and extend the existing pattern: LEGAL CANDIDATES -> COMPACT
RELEVANT CONTEXT -> LLM -> PARSE -> VALIDATE -> APPLY THROUGH ENGINE ->
LOG -> PERSIST.

======================================================================
3. TARGET HIERARCHY
======================================================================

Recommended conceptual architecture:

TOWN MIND -> Environment & Ecology Director -> Social Dynamics Director
-> NPC Life Director -> Economy & Business Director -> Traffic/Visitors
Director -> Event & Opportunity Director -> Narrative Opportunity
Director -> Horror Director -> Core NPC Minds -> Batched Lightweight
Resident Simulation -> Dialogue Model -> Memory/Summary Model THEN:
VALIDATORS & DOMAIN MODELS -> DETERMINISTIC STATE APPLICATION ->
PERSISTENT STATE

This is conceptual. A low-end machine may route all roles to one model.
Better hardware may assign specialist models.

The Town Mind is invisible to the player. It is NOT the fictional
Chorus. Shared infrastructure is acceptable, but simulation authority
and fictional supernatural authority must remain separate.

======================================================================
4. TOWN MIND
======================================================================

The Town Mind is a long-horizon strategic layer. It receives compact
summaries of village state, important recent events, player patterns,
social tensions, NPC concerns, economic pressure, environment,
unresolved procedural arcs, story phase, horror eligibility, pacing
history, and recurrence-permitted previous-run patterns.

It may store revisable intentions such as: - allow two quiet days before
another anomaly opportunity; - support a developing conflict without
forcing its outcome; - create opportunities for an isolated NPC to
reconnect; - increase bakery supply pressure within economic rules; -
let Mara’s ecological suspicion develop if evidence supports it; -
surface a neglected system gently; - avoid recently repeated event
families.

Intentions are not scripts. Player actions, NPC actions, simulation
changes, and validators may invalidate them. The Town Mind must revise
plans rather than force them.

======================================================================
5. CLOCKS, PRIORITY, AND ASYNC WORK
======================================================================

Do not call strategic AI after every minor action.

DETERMINISTIC TICK: every relevant time advance; growth arithmetic,
timers, recovery, obligations, legal progression. DOMAIN PULSE: bounded
in-game intervals or state changes; traffic, NPC candidate choice,
social opportunities, environment. TOWN MIND REVIEW: morning/evening,
major revelation, relationship transition, business crisis,
failure/recovery, arrival, arc ending, several in-game hours, prolonged
low novelty, or major playstyle shift.

Recommended priority: 1. player dialogue 2. urgent NPC decision 3. major
event resolution 4. Town Mind update 5. social simulation 6.
environment/ecology 7. ambient flavour

Async-capable: Town Mind reviews, environmental tendencies, background
social simulation, resident batches, summarization, procedural arc
proposals, economic trends, visitor planning, long-horizon intentions.

Normally synchronous when player waits: free dialogue, immediate
reaction, visible urgent decision, immediate confrontation resolution.

Every async proposal must include source state revision/time,
assumptions, expiry conditions, and validation schema. Revalidate before
application. Never apply stale AI output blindly.

Quiet periods are mandatory. A good Director must know when to do
nothing.

======================================================================
6. MULTI-MODEL CAPABILITY TIERS
======================================================================

Logical roles: 1. Strategic reasoning model — Town Mind and complex
planning. 2. Structured simulation model — bounded choices and schema
proposals. 3. Dialogue/roleplay model — in-character free conversation.
4. Memory/summary model — compression, salience, retrieval assistance.

LOW-END: one small local model fills all roles; calls serialized,
batched, prioritized. MID-TIER: dialogue and strategy may differ; small
fast model handles structured work. HIGHER-TIER: specialists assigned
independently.

The current low-end target remains supported. Future preferred hardware
may be a more capable SFF/ThinkCentre-class machine. Do not architect
around one model family or require cloud GPUs.

Every task declares: role, urgency, sync/async, context budget,
generation budget, output contract, validator, retry/repair policy,
fallback, and authority level.

======================================================================
7. ENVIRONMENT AND ECOLOGY
======================================================================

LLMs may influence weather character/tendency, unusual sequences,
seasonal character, vegetation patterns, flowering/growth anomalies,
animals, vehicles, traffic, environmental mood, and eligible
supernatural environmental pressure.

Use two layers: STRATEGIC TENDENCY: “wet unsettled week”, “early
flowering”, “fox activity rising”, “delivery traffic disrupted”. BOUNDED
MOMENT CHOICE: legal weather transition, animal sighting, vehicle
arrival, localized detail.

Calendar progression is authoritative. LLMs cannot jump arbitrarily
between seasons. They influence the character of a season and valid
abnormalities.

Ecology must respect calendar, biology, existing vegetation, weather
history, location, and supernatural eligibility. No arbitrary instant
transformation.

======================================================================
8. NPC COGNITION AND AUTONOMY
======================================================================

Core/semi-main NPCs should have persistent structured internal state: -
concerns; - short- and long-term goals; - ambitions; - unresolved
questions; - plans; - interpretations of events; - beliefs; -
suspicions; - commitments; - relationship impressions; - anomaly
theories; - confidence; - salient memories; - desired interactions; -
avoidance targets.

LLMs propose updates; engine validates and stores them. Internal
thoughts are not normally shown directly.

Within constraints, AI-guided NPC development may include interests,
hobbies, job dissatisfaction/change, friendship, conflict,
reconciliation, loneliness, ambition, investigation, avoidance,
attraction, loss of interest, jealousy, visits, invitations, changing
routines, and evolving mystery theories.

Knowledge boundaries are mandatory. An NPC may infer from evidence they
possess; they may not acquire hidden canon because the model knows it.

======================================================================
9. FACT, KNOWLEDGE, BELIEF, SUSPICION, CLAIM, AND LIE
======================================================================

State should distinguish: OBJECTIVE FACT — authoritative truth.
KNOWLEDGE — what an NPC actually learned/witnessed. BELIEF — what they
think is true. SUSPICION — possibility under consideration. CLAIM — what
they said. DECEPTION — claim knowingly contrary to their belief.
CONFIDENCE — strength. SOURCE — why they know/believe it.

This enables lies, mistakes, gossip, accusations, conflicting theories,
false memories, and correction without corrupting canon.

If the player falsely says “Mara told me you stole from the shop”, the
system may extract the claim and attribution, assess trust and existing
knowledge, create suspicion/disbelief, remember it, cause confrontation
or gossip later, expose the lie, and change trust. The prose itself
never mutates state.

======================================================================
10. FREE DIALOGUE AND CONSEQUENCES
======================================================================

NPCs may engage in natural small talk and discuss their fictional world,
work, relationships, residents, observed events, institutions, plausible
interests, known evidence, beliefs, and suspicions.

If the player raises specialist outside topics not grounded in the
world, NPCs respond naturally but shallowly according to character. A
baker must not become a quantum-cosmology expert because the pretrained
model can answer.

Dialogue prompts must specify identity, competence, knowledge boundary,
beliefs, relationship, relevant memory, current situation, and forbidden
knowledge.

Conversation may create structured consequences, initially limited to
high-confidence acts: - explicit promise; - invitation; - threat; -
insult; - apology; - disclosed secret; - factual claim; - request; -
agreement; - refusal.

Pipeline: PLAYER TEXT -> DIALOGUE RESPONSE -> SEPARATE STRUCTURED
INTERPRETER -> CANDIDATE SOCIAL ACTS -> VALIDATORS -> BOUNDED STATE
EFFECTS -> MEMORY -> FUTURE PROPAGATION.

Example: “I’ll bring apples tomorrow” may become a structured promise
only if item/category, recipient, deadline, and explicitness validate.

Do not mechanize every sentence. Ambiguous conversation remains
conversation.

Dialogue style evolves from specific history, not only a relationship
threshold.

======================================================================
11. PROCEDURAL MULTI-DAY ARCS
======================================================================

LLMs may create multi-day emergent arcs by composing validated
primitives, not by writing unconstrained prose quests and forcing the
engine to enact them.

Example: validated supply delay -> bakery production falls -> Jonah
works longer -> availability changes -> another NPC becomes concerned ->
player opportunity -> state-dependent resolution.

Possible arc families: business pressure, staffing, delivery problem,
local disagreement, missing ordinary object, event preparation,
friendship conflict, hobby, job consideration, visitor, community
project, ecological concern, repair need.

Procedural arcs must be internally distinct from authored quests even if
UI presentation is natural. Persistent consequences are allowed after
validation.

======================================================================
12. LIGHTWEIGHT GENERATED RESIDENTS
======================================================================

Main and semi-main NPCs are authored. Approximately 20–25 very
lightweight background residents may be generated once at run start
under strict schema and validators.

Possible fields: setting-appropriate name, age band, household relation,
legal occupation, home area, few traits, ordinary interests, initial
social links, speech tendency, schedule archetype.

They persist for the run and may accumulate job changes, friendships,
disputes, rumours, reputation, event participation, and minor history.

Do not invent residents opportunistically mid-conversation to support
hallucinated claims. Generated residents cannot overwrite authored cast
or canon. Cross-run handling should later be explicitly integrated with
recurrence.

======================================================================
13. ECONOMY, JOBS, AND OFFSCREEN LIFE
======================================================================

LLMs may propose supply shortages, hiring, reduced capacity, farm
performance, business conflict, schedule change, opportunities, and job
dissatisfaction.

Pattern: CAUSE -> LEGAL PROPOSAL -> DOMAIN VALIDATOR -> BOUNDED EFFECT
-> ENGINE APPLICATION -> DOWNSTREAM CONSEQUENCES.

LLMs cannot directly edit money, stock, employment, or business health.

The village continues offscreen. If the player farms for ten days,
friendships, businesses, rumours, theories, visitors, jobs, conflicts,
reconciliations, and ecological changes may occur elsewhere. The player
learns through behaviour, gossip, noticeboards, business state,
conversation, environment, and records.

The player matters but is not the only active person.

======================================================================
14. PACING, BOREDOM, AND ADAPTATION
======================================================================

The Town Mind observes farming focus, social focus, investigation,
economic grind, avoidance, repeated routine, story rushing, retreat into
ordinary life after horror, low novelty, and excessive event density.

It adapts opportunities and pressure: - farming focus: agricultural
opportunities, produce interactions, ecological clues, slower mystery
pressure; - investigation focus: clue opportunities, defensive
reactions, institutional friction, exhaustion risk; - social focus:
interpersonal consequence, gossip, contradictory testimony,
relationship-mediated horror.

It may occasionally create tension around preferred activities but
should not constantly punish playstyle.

If several days pass without meaningful novelty, it may propose a legal
event, visitor, social complication, opportunity, or anomaly. It must
also deliberately preserve quiet periods.

======================================================================
15. ADAPTIVE HORROR
======================================================================

The Horror Director may adapt legally to behaviour: - routine dependence
-> routine corruption; - social dependence -> contradictory testimony,
gossip, synchronization; - investigation -> record instability, fair
false leads, defensive anomalies; - isolation ->
environmental/perceptual pressure; - economic focus -> uncanny
discrepancies in exchange; - relationship dependence -> misrecognition
or contradictory memory.

Limits: - no real-world sensitive personal data; - no cheap personalized
punishment; - no constant negation of preferred play; - only canonical
anomaly powers; - eligibility and fair clue rules remain; - playability
remains unless legitimate failure occurs.

The Horror Director may reason from the Chorus’s drives but cannot seize
arbitrary Town Mind or engine authority.

======================================================================
16. LEARNING THE PLAYER
======================================================================

“Learning” means structured memory, retrieval, summaries, impressions,
and adaptation—not fine-tuning model weights.

Maintain distinct models:

A. OBJECTIVE BEHAVIOURAL PROFILE Engine-derived frequencies and facts:
activities, conversation, investigation, relationships, risk, promises,
avoidance, economy, routines.

B. TOWN MIND MODEL Revisable strategic interpretation: pacing
preference, neglected systems, engagement pattern, tendencies.

C. NPC-SPECIFIC IMPRESSIONS Each NPC’s personal view based only on
interaction, knowledge, gossip, and memory.

D. CHORUS MODEL Fictional bounded model based only on patterns the
supernatural system has canonically had opportunity to learn.

These may disagree and may be wrong. They should revise with evidence.
This distinction is thematically important.

======================================================================
17. LONG-TERM MEMORY AND CROSS-RUN ADAPTATION
======================================================================

Recent prompt history is insufficient. Important events from 40+ game
days ago must be retrievable when salient.

Recommended memory layers: EVENT STORE — actors, place, time, type,
consequence, witnesses, importance, emotion, unresolved links. NPC
MEMORY — experienced or learned memories. RELATIONSHIP MEMORY —
important shared history. ARC MEMORY — cause, stages, participants,
resolution. SUMMARY MEMORY — compressed periods. RUN MEMORY — compressed
previous-run history and player patterns.

Before a call, retrieve by actor, topic, location, relationship,
unresolved status, recency, and salience. Include only the minimum
useful subset. Never solve memory by stuffing the entire transcript into
prompts.

Cross-run memory may retain compressed patterns such as repeatedly
trusting Mara, avoiding clinic, rushing investigation, farming after
fear, breaking promises, relying on a route, or forming similar
relationships. Use subtly for echoes and adaptation, never unfair
meta-punishment.

======================================================================
18. HALLUCINATION PREVENTION — NON-NEGOTIABLE
======================================================================

Do not rely on “do not hallucinate.” Use layered control:

1.  MINIMAL TASK CONTEXT — only relevant facts.
2.  POSITIVE GROUNDING — legal actors, places, items, relationships,
    facts, actions, history.
3.  NEGATIVE AUTHORITY BOUNDARY — explicit forbidden inventions.
4.  CLOSED ENUMERATIONS — IDs/options/enums where possible.
5.  STRUCTURED OUTPUT — schema for state proposals.
6.  PARSER — reject malformed output.
7.  SEMANTIC VALIDATOR — existence, knowledge, geography, chronology,
    causality, resources, canon, authority.
8.  PRECONDITION VALIDATION — assumptions still true at application.
9.  REPAIR — compact correction prompt containing validation errors and
    legal boundaries.
10. BOUNDED RETRIES.
11. FALLBACK — deterministic or simpler bounded mode.
12. HEALTH/AUTHORITY MANAGEMENT.
13. AUDIT LOG.

The LLM is never trusted because prose sounds plausible.

======================================================================
19. PROMPT ENGINEERING STANDARD
======================================================================

Every production prompt template should contain: ROLE — who it acts as.
TASK — one primary task. AUTHORITY — what it may decide. NON-AUTHORITY —
what it cannot decide. WORLD FACTS — relevant authoritative facts only.
KNOWLEDGE BOUNDARY — exactly what actor knows/believes. CURRENT STATE —
relevant structured state. MEMORY — retrieved salient memories only.
LEGAL VOCABULARY — IDs/options/entities/actions. OUTPUT CONTRACT — exact
schema or response form. FAILURE INSTRUCTION — return NONE/NO_CHANGE
instead of inventing.

Bad: “What should happen next in Bellwether?”

Good principle: “You are the Social Dynamics Director. Choose zero or
one legal development from the supplied candidates. Do not create
residents, places, facts, deaths, jobs, or relationships outside the
supplied state. Return the required schema. If no option is causally
appropriate, return NO_CHANGE.”

Prompts must be task-specific, compact, explicit, versioned,
benchmarked, and adversarially tested.

======================================================================
20. VALIDATOR FAMILIES
======================================================================

Required validators: - entity existence; - geography/movement; - time,
schedule, deadline, season; - NPC knowledge; - belief-change cause; -
social bounds and causality; - economy/resources/jobs; - event
prerequisites and conflicts; - authored story/canon; - horror
eligibility and fairness; - danger catalogue and cause; - recurrence
permissions; - dialogue consequences; - core-cast permanent consequence
safety; - async staleness.

Validators are part of game design, not optional defensive code.

======================================================================
21. HEALTH AND AUTHORITY DEGRADATION
======================================================================

Track per model/role: success, timeout, empty output, parse failure,
semantic validation failure, repair success, retries, fallback rate,
latency, stale async results, repeated choice bias, context size, and
token exhaustion.

Authority degradation is granular: LEVEL 3 — compose novel validated
proposal. LEVEL 2 — choose generated candidates and optionally justify.
LEVEL 1 — closed-option choice only. LEVEL 0 — deterministic fallback
for that task until recovery check.

A model failing economy output should not disable dialogue. The system
should continue health/recovery checks rather than permanently
abandoning AI after one failure.

The intention is to use LLMs wherever designed while protecting the game
through prompts, validators, repair, and controlled authority.

======================================================================
22. AUDITABILITY
======================================================================

Developer mode should log every consequential AI decision: - real and
game timestamp; - task ID; - model; - role; - prompt template version; -
compact context summary; - retrieved memory IDs; - candidate IDs; -
retained raw output; - parsed proposal; - validation result/errors; -
repair attempts; - fallback; - accepted decision; - state delta; -
latency; - length/token information where available.

Ordinary players never see Director internals.

======================================================================
23. NOVEL COMBINATIONS AND EMERGENCE
======================================================================

The Town Mind and Directors are explicitly allowed to propose
combinations developers did not individually script, provided every
component and consequence is legal.

Developers define entities, systems, actions, event primitives,
constraints, causal rules, canon, and validators. The LLM composes them.

Example: delivery delay -> Jonah misses event -> existing disagreement
worsens -> another resident asks player for help -> rain damages
unattended garden task.

If every link is legal, causal, validated, and persistent, the
unanticipated combination is allowed.

Emergence means unanticipated legal composition, not unvalidated
invention.

======================================================================
24. ALLOWED AND FORBIDDEN AUTHORITY
======================================================================

LLMs MAY influence, through validators: strategic attention, pacing,
quiet periods, weather transitions, environmental tendencies, ecological
observations, animals/vehicles, NPC actions, concerns/plans,
beliefs/suspicions, social opportunities, gossip choices, conversation
prose, bounded conversation consequences, procedural arcs, economic
trends, business pressure, job consideration, visitors, minor events,
player-style adaptation, legal horror pressure, recurrence echoes,
summaries, and retrieval.

LLMs MAY NEVER DIRECTLY: - mutate arbitrary state; - rewrite saves; -
create canon by assertion; - change metaphysics; - reveal undiscovered
truth; - invent impossible knowledge; - create unvalidated locations; -
teleport actors; - create money/items from nothing; - break calendar
season progression; - kill/remove core NPCs without gates; - complete
authored quests without engine conditions; - select ineligible
endings; - turn lies into objective truth; - overwrite authored story
with procedural arcs; - bypass danger/horror/recurrence validators; -
apply stale async results blindly; - use sensitive real-world user data
for horror personalization.

======================================================================
25. DETERMINISTIC FALLBACK
======================================================================

Bellwether remains fully playable without the LLM, though less dynamic.

Fallback must preserve travel, time, jobs, economy, gardening, cooking,
relationships, authored dialogue, investigation, story, horror
eligibility, danger, recurrence, save/load, legal NPC routines, weather,
and basic traffic/events.

Fallback hierarchy: 1. safe context-equivalent cached result where
valid; 2. deterministic candidate choice; 3. weighted local simulation;
4. authored fallback prose; 5. NO_CHANGE when safest.

Free dialogue fallback may be limited but must never corrupt state.

======================================================================
26. PERFORMANCE PHILOSOPHY
======================================================================

Use LLMs for interpretation, composition, adaptation, planning,
language, and summarization.

Do not use them for crop arithmetic, inventory counts, adjacency checks,
calendar advancement, price arithmetic, quest prerequisites, or entity
existence.

The engine does what code does best. The LLM does what contextual
reasoning and language do best.

======================================================================
27. IMPLEMENTATION ROADMAP
======================================================================

PHASE A — Harden current AI: inventory calls, version prompts, task
metadata, traces, role health, schemas, hallucination tests. PHASE B —
Memory: event store, NPC memory, relationship memory, salience,
retrieval, summaries, cross-run patterns. PHASE C — NPC cognition:
concerns, goals, plans, beliefs, suspicions, claims, deception,
commitments. PHASE D — Conversation consequences: promises, claims,
secrets, invitations, insults, apologies, validators. PHASE E — Town
Mind: strategic state, intention schema, pacing, quiet periods, novelty
detection, revision/expiry. PHASE F — Procedural arcs: primitive
library, causal graph, multi-day schemas, social/economic/environmental
integration. PHASE G — Specialist routing: configurable roles,
capability tiers, benchmarks, graceful one-model mode. PHASE H — Async
scheduler: priority queue, revisions, stale checks, background strategic
and summary work. PHASE I — Lightweight residents: strict schema,
run-start generation, persistence, batched simulation. PHASE J —
Adaptive horror and cross-run learning: separate player models, Chorus
constraints, recurrence-aware patterns, fair adaptive pressure.

======================================================================
28. ACCEPTANCE TESTS
======================================================================

The system is successful only if: 1. Divergent playstyles produce
materially different Day-30 village histories. 2. Both remain
canonically legal. 3. NPCs never know undiscovered facts without valid
source. 4. Lies never alter objective truth. 5. Old salient events can
be retrieved naturally. 6. Generated arcs persist across days. 7.
Offscreen events produce later observable consequences. 8. Quiet periods
occur intentionally. 9. Low novelty can be detected without event spam.
10. Conversation can create validated promises and memories. 11. Invalid
outputs never corrupt state. 12. Repeated failures reduce authority
visibly in developer mode. 13. Game remains playable with Ollama
offline. 14. Stale async proposals are rejected. 15. Seasons follow
calendar progression. 16. Environmental abnormalities respect
eligibility. 17. Core NPCs cannot be casually killed by AI. 18.
Lightweight residents persist consistently. 19. Cross-run adaptation is
subtle rather than punitive. 20. One small model can fill all roles. 21.
Specialist models can be assigned independently. 22. Every consequential
accepted AI decision is auditable. 23. Novel combinations occur that
were not individually scripted. 24. Those combinations use only legal
primitives. 25. Dialogue respects character competence and knowledge
boundaries.

======================================================================
29. FINAL DESIGN STATEMENT
======================================================================

Bellwether’s LLM system is not a decorative chatbot attached to a
deterministic game. It is a constrained hierarchical simulation
intelligence that helps the village observe itself, form intentions,
change socially, adapt environmentally, remember the player, create
legal multi-day developments, and respond differently to different
lives.

The deterministic engine protects reality. Authored canon protects
meaning. Validators protect coherence. Memory protects continuity.
Prompt engineering protects boundaries. The LLM supplies interpretation,
adaptation, composition, and surprise.

Too little LLM authority makes the village repetitive. Too much
unvalidated authority makes it incoherent.

The target is high emergence inside hard truth:

SURPRISING OUTCOMES. LEGAL CAUSES. PERSISTENT CONSEQUENCES. AUDITABLE
DECISIONS. NO HALLUCINATED REALITY.

======================================================================
END OF LLM ROLE, AUTHORITY, AND CONSTRAINT SPECIFICATION

BELLWETHER — CODING AND IMPLEMENTATION PHILOSOPHY Version 1.0 |
2026-07-08 | Authoritative engineering philosophy

PURPOSE

This document defines how Bellwether should be developed, modified,
tested, extended, debugged, and released. It exists to prevent future
development sessions, coding agents, LLMs, or human contributors from
damaging working systems, replacing established architecture casually,
or implementing features in ways that undermine the game’s long-term
design.

Use together with: - Bellwether Comprehensive Project Context v2 -
Bellwether Engineering Context Snapshot - Bellwether Canonical World and
Story Bible v1 - Bellwether LLM Role, Authority, and Constraint
Specification v1

This document is not a substitute for reading the current codebase.
Before modifying Bellwether, inspect the actual current implementation.
Documentation describes intent; the repository defines the exact
implementation state.

====================================================================== 1.
PRIMARY ENGINEERING PRINCIPLE
======================================================================

BELLWETHER MUST EVOLVE ADDITIVELY.

Existing working systems are assets, not disposable prototypes.

Do not casually: - rewrite working modules; - replace established
systems with simplified versions; - delete behaviour because a new
feature is easier without it; - collapse complex state into temporary
shortcuts; - remove compatibility paths without migration; - overwrite
authored content; - bypass persistence; - replace data-driven content
with hard-coded logic; - regress existing simulation to make a new
feature work.

The preferred development pattern is:

INSPECT -> UNDERSTAND -> MAP DEPENDENCIES -> DEFINE INVARIANTS -> ADD OR
EXTEND -> MIGRATE IF NECESSARY -> TEST OLD BEHAVIOUR -> TEST NEW
BEHAVIOUR -> PLAYTEST -> PACKAGE A COMPLETE RUNNABLE BUILD

A feature is not complete merely because its new path works. It is
complete when the new path works and previously accepted behaviour still
works.

======================================================================
2. THE ACCEPTED BASELINE IS FROZEN AS BEHAVIOURAL CONTRACT
======================================================================

The accepted v0.1.0 baseline and all subsequently accepted working
releases form a behavioural contract.

Future development must preserve established working behaviour across,
where applicable: - simulation; - conversations; - persistence; -
Director systems; - weather; - seasons; - story; - horror; -
relationships; - investigation; - branching; - recovery; - recurrence; -
endgame; - testing; - executable permissions; - UI behaviour; - map
systems; - asset lookup; - LLM fallback.

“Frozen” does not mean code can never change. It means behaviour cannot
be removed or regressed casually.

Refactoring is permitted only when: - the existing behaviour is
understood; - tests capture the behaviour; - migration is planned; -
compatibility is verified; - the refactor has a concrete engineering
benefit.

Do not rewrite for aesthetic preference alone.

======================================================================
3. READ BEFORE WRITING
======================================================================

Before implementing any significant feature:

1.  Inspect repository structure.
2.  Locate relevant models, services, UI code, content data, tests,
    persistence code, and entry points.
3.  Search for all callers of code to be modified.
4.  Search for serialized field names and save compatibility
    implications.
5.  Identify existing extension points.
6.  Identify current fallback behaviour.
7.  Identify tests and missing test coverage.
8.  Run the current game before modification when possible.
9.  Record behavioural invariants.
10. Only then design the patch.

Never infer architecture from filenames alone.

Never assume a system is absent because its UI exposure is incomplete.

Never build a duplicate system before checking whether the existing
system already implements part of the requested behaviour.

======================================================================
4. THE ENGINE OWNS TRUTH
======================================================================

Bellwether is state-driven.

Authoritative state belongs in deterministic models and persistent
storage, not in generated prose, UI text, prompt history, or temporary
local variables.

Examples: - inventory is authoritative state; - relationship values and
structured memories are authoritative state; - quest completion is
authoritative state; - NPC location is authoritative state; - calendar
and season are authoritative state; - economy and job state are
authoritative state; - known evidence is authoritative state; -
recurrence history is authoritative state.

Generated language may describe state. It must not become the only place
state exists.

The UI renders state. The LLM interprets and proposes changes. The
engine validates and applies changes. Persistence stores them.

======================================================================
5. DOMAIN MODELS BEFORE UI SHORTCUTS
======================================================================

When adding a mechanic, implement or extend its domain model before
embedding behaviour directly into the UI.

Preferred: PLAYER ACTION -> DOMAIN/SERVICE METHOD -> VALIDATION -> STATE
CHANGE -> EVENT/RESULT -> UI RENDER

Avoid: BUTTON HANDLER -> DIRECTLY EDIT FIVE UNRELATED FIELDS -> PRINT
RESULT

UI code should coordinate presentation and input, not become the hidden
game engine.

If a feature initially needs a narrow UI implementation, isolate it
behind a clear service or method so it can later be expanded.

======================================================================
6. DATA-DRIVEN CONTENT
======================================================================

Bellwether should remain data-driven where content varies by: -
character; - location; - season; - weather; - time of day; - story
phase; - anomaly state; - relationship state; - event; - asset variant.

Prefer manifests, registries, catalogues, schemas, and content files
over sprawling hard-coded conditional chains.

However, do not move logic into data merely to claim the system is
data-driven. Rules belong in code. Content and configuration belong in
data.

Good division: - code defines how weather transitions are legal; - data
defines weather descriptions and assets; - code defines quest state
transitions; - data defines authored quest content; - code defines NPC
scheduling rules; - data defines NPC schedule preferences.

Validate data at startup or in tests. Fail clearly in developer mode.

======================================================================
7. SAVE COMPATIBILITY AND MIGRATION
======================================================================

Persistence is a core game system.

Every state change must consider: - new game; - current save; - old
save; - recurrence; - partial state; - missing optional fields; -
invalid/corrupt state; - version migration.

When adding fields: - provide safe defaults; - version state when
necessary; - migrate explicitly; - preserve unknown compatible data
where practical; - test loading at least representative prior saves.

Never solve a migration problem by silently wiping player state.

Recovery/reset systems require special care: reset only the state
categories the design says should reset. Story milestones and town
continuity must not be accidentally erased by broad object
reinitialization.

======================================================================
8. LLM INTEGRATION PHILOSOPHY
======================================================================

LLM integration must follow the dedicated LLM specification.

The short engineering rule is:

THE LLM PROPOSES. VALIDATORS DECIDE LEGALITY. THE ENGINE APPLIES. THE
SAVE STORES.

Never let generated prose directly mutate state.

Every consequential LLM task should have: - explicit role; - compact
context; - authority boundary; - legal vocabulary; - output contract; -
parser; - semantic validator; - retry/repair policy; - deterministic
fallback; - trace logging.

Do not solve hallucination by merely making prompts longer. Better
grounding, narrower authority, structured output, entity IDs,
validators, and retrieval are preferred over giant context dumps.

======================================================================
9. SMALL MODELS ARE A DESIGN CONSTRAINT
======================================================================

Bellwether currently targets local operation on low-end CPU hardware and
should scale upward.

Therefore: - keep prompts compact; - avoid repeated full-world
context; - batch background work; - cache only when semantically safe; -
use deterministic code for arithmetic and legality; - prefer bounded
choice tasks for weak models; - serialize calls when local hardware
requires it; - support asynchronous background work where causally
safe; - prioritize foreground conversation; - make logical AI roles
configurable independently of physical models.

Do not require a large cloud model for core playability.

Better hardware should increase richness, latency tolerance, and
specialist routing—not change the fundamental legality of the game.

======================================================================
10. PERFORMANCE BUDGETING
======================================================================

Every new system should consider: - CPU cost; - memory cost; - disk
I/O; - prompt size; - LLM call frequency; - UI responsiveness; - save
size; - simulation scaling with NPC count.

Do not optimize blindly, but do not introduce obvious per-tick expensive
work when an event-driven or batched design is available.

Prefer: - dirty flags; - event-driven updates; - scheduled pulses; -
cached derived values; - bounded histories; - indexed retrieval; -
batched lightweight NPC simulation.

Measure before major optimization.

A low-end machine should remain a first-class test target.

======================================================================
11. SEPARATION OF CLOCKS
======================================================================

Do not make every system update at the same frequency.

Bellwether should distinguish: - immediate player action resolution; -
deterministic time advancement; - NPC schedule updates; - environment
pulses; - social simulation pulses; - strategic LLM reviews; - daily
maintenance; - seasonal/calendar progression; - recurrence transitions.

This improves performance, causality, and debugging.

A system should update when its state can meaningfully change, not
merely because a render loop ran.

======================================================================
12. EVENT-DRIVEN CAUSALITY
======================================================================

Persistent changes should have traceable causes.

Preferred: SUPPLY DELAY EVENT -> BUSINESS PRESSURE -> SCHEDULE CHANGE ->
MISSED SOCIAL EVENT -> RELATIONSHIP CONSEQUENCE -> PLAYER OPPORTUNITY

Avoid arbitrary state mutation such as: “relationship decreased because
Director wanted drama.”

Causal chains make emergence understandable, testable, and fair.

Events should carry enough metadata to explain: - cause; - actors; -
location; - time; - witnesses; - state delta; - downstream eligibility.

======================================================================
13. NPC ENGINEERING PHILOSOPHY
======================================================================

Core NPCs are persistent agents, not dialogue vending machines.

Their implementation should distinguish: - authored identity; - stable
traits; - mutable needs; - goals; - beliefs; - knowledge; - memories; -
relationships; - schedules; - commitments; - current concerns; - current
plans; - recurrence sensitivity.

Do not store every aspect as free text.

Use structured state for things the engine must reason about. Use
generated language for expression and interpretation.

Core NPCs receive richer simulation. Semi-main NPCs receive moderate
simulation. Lightweight residents receive cheaper batched simulation.

Do not apply identical computational cost to every resident.

======================================================================
14. AUTHORED STORY AND EMERGENT SIMULATION MUST COEXIST
======================================================================

Bellwether has fixed authored story architecture and emergent
simulation.

Neither should erase the other.

Authored story owns: - canonical revelations; - major mystery facts; -
key quest gates; - true ending logic; - protected character outcomes; -
metaphysics.

Emergent simulation owns, within bounds: - timing opportunities; -
social context; - procedural arcs; - minor conflicts; - business
pressures; - environmental developments; - NPC interpretations; -
offscreen life; - player-specific pacing.

The emergent layer should create different paths through a stable truth.

Do not procedurally generate the central mystery.

Do not make authored quests freeze the whole simulation unnecessarily.

======================================================================
15. UI PHILOSOPHY
======================================================================

The UI should remain readable, responsive, and uncluttered.

Principles: - show what the player needs now; - allow deeper information
through dedicated views; - avoid exposing raw simulation internals; -
preserve atmosphere; - avoid debug terminology in player UI; - make
important state discoverable; - do not require memorizing terminal
output; - prevent decorative elements from obscuring functional
information; - treat different window sizes as real supported
conditions.

Map overlays, legends, compass elements, portraits, text panels, and
controls must be tested together at actual runtime dimensions.

A UI element being present in code does not mean it is visible or
usable.

======================================================================
16. ASSET PHILOSOPHY
======================================================================

Heavy visual assets may live outside lightweight source distributions
when the project workflow requires it. Code must use stable asset paths
and manifests.

Do not: - embed large binary assets into source files; - duplicate heavy
assets unnecessarily; - silently rename asset keys; - break existing
manifests; - assume missing optional art should crash the game.

Provide graceful fallback for absent optional assets. Required assets
should fail clearly in developer/testing mode.

Portraits should remain fixed for now unless the design explicitly
changes. Avoid unnecessary asset explosion.

Background variants should use consistent scene identity across
weather/season variants rather than unrelated compositions.

======================================================================
17. ERROR HANDLING
======================================================================

Player-facing mode should fail gracefully. Developer mode should fail
informatively.

Do not swallow exceptions silently.

Use: - clear logs; - contextual error messages; - safe fallbacks; -
state rollback where partial application is dangerous; - atomic save
patterns where practical.

For multi-step state changes: VALIDATE ALL PRECONDITIONS -> PREPARE
DELTA -> APPLY COHERENTLY -> PERSIST -> EMIT RESULT

Avoid half-applied event consequences.

======================================================================
18. LOGGING AND OBSERVABILITY
======================================================================

An emergent simulation must be inspectable.

Developer logs should make it possible to answer: - why did this NPC do
this? - what caused this relationship change? - why did this event
become eligible? - what did the LLM propose? - why was it rejected? -
what fallback ran? - what state changed? - what memory was retrieved? -
why did recovery occur here?

Logs should use stable categories and identifiers.

Do not make ordinary players read logs to understand normal gameplay.

======================================================================
19. TESTING PYRAMID
======================================================================

Bellwether requires several complementary test layers.

A. UNIT TESTS Pure rules, validators, calculations, parsing, migrations.

B. INTEGRATION TESTS Systems interacting: jobs/economy,
conversation/memory, event/social state, recurrence/persistence,
map/travel.

C. DETERMINISTIC SIMULATION TESTS Long state progressions under seeded
or controlled choices.

D. LLM CONTRACT TESTS Prompt output parsing, malformed output,
hallucinated entities, forbidden knowledge, repair, fallback, authority
degradation.

E. SAVE COMPATIBILITY TESTS Representative old and current saves.

F. UI SMOKE TESTS Launch, navigation, map, overlays, portraits, text
flow, resizing where applicable.

G. FULL PLAYER-PERSPECTIVE PLAYTESTS Mandatory before meaningful
releases.

No single layer replaces the others.

======================================================================
20. PLAYER-PERSPECTIVE PLAYTESTING
======================================================================

Before release, simulate full playthroughs as a player, not merely as a
developer who knows the code.

Read prose and choices in sequence.

Test multiple styles: - social; - investigation-focused; -
life-sim/farming; - economic/job-focused; - cautious; - reckless; -
story-avoiding; - mixed.

Assess: - whether the story is understandable without code knowledge; -
whether choices are motivated; - whether pacing works; - whether
life-sim activity exists before horror escalation; - whether horror
escalation is legible; - whether investigation is fair; - whether
failure and recurrence make sense; - whether endings are reachable; -
whether systems remain usable after long play; - whether repetition
becomes tedious; - whether the town feels alive.

The developer should not rely only on deterministic ending simulations.

The user intends to discover the story as a player and should not need
to spoil themselves to verify quality.

======================================================================
21. REGRESSION TESTING
======================================================================

Every feature release should include a regression checklist covering
affected and adjacent systems.

Example: a map release should test: - movement; - player marker; -
region reveal; - route reveal; - parchment mask; - legend visibility; -
compass behaviour; - resize; - travel state; - save/load; - non-map UI.

A portrait release should test: - portrait lookup; - missing portrait
fallback; - layout; - text wrapping; - NPC conversation; - asset
manifest.

Think in dependency neighbourhoods, not isolated files.

======================================================================
22. RELEASE PHILOSOPHY
======================================================================

A release candidate should be: - runnable; - versioned; - internally
consistent; - tested; - packaged with required source changes; - free of
temporary debug hacks; - accompanied by concise change notes; - checked
for executable permissions where relevant.

Do not call an implementation complete if the user must manually merge
code fragments, infer file placement, or repair formatting.

The desired delivery unit is a complete runnable build or complete
replacement files with exact paths.

When packaging is available, provide the complete package. When only
file delivery is possible, provide complete files, not illustrative
snippets.

======================================================================
23. CODING AGENT WORKFLOW
======================================================================

A coding agent working on Bellwether should follow:

STEP 1 — inspect repository and context documents. STEP 2 — restate
internal implementation target. STEP 3 — identify exact files and
systems affected. STEP 4 — inspect all relevant code before editing.
STEP 5 — define invariants and compatibility requirements. STEP 6 —
implement the smallest coherent architectural change. STEP 7 —
add/update tests. STEP 8 — run targeted tests. STEP 9 — run full suite.
STEP 10 — run game smoke test. STEP 11 — conduct relevant
player-perspective test. STEP 12 — inspect diff for accidental deletions
or simplifications. STEP 13 — package runnable output. STEP 14 —
document version and changes.

Do not ask the user to manually edit code when the agent has the
capability to modify and return complete files.

Do not give pseudo-code when production code was requested.

Do not replace a requested implementation with instructions for the user
to implement it.

======================================================================
24. PATCH SIZE AND REFACTORING
======================================================================

Prefer the smallest coherent patch, not necessarily the fewest changed
lines.

A coherent patch: - has one clear purpose; - uses existing
architecture; - includes necessary migration; - includes tests; - avoids
unrelated rewrites.

Large refactors should be separated from feature additions when
practical.

If a refactor is required for the feature: - document why; - preserve
behaviour; - test before and after; - avoid opportunistic cleanup of
unrelated modules.

======================================================================
25. NAMING AND SCHEMA STABILITY
======================================================================

Use stable identifiers for: - NPCs; - locations; - items; - jobs; -
events; - quests; - story beats; - assets; - LLM task types.

Display names may change independently of IDs.

Do not use mutable display strings as persistence keys when stable IDs
are possible.

Schema changes should be versioned and migrated.

LLM prompts should use stable IDs plus human-readable labels where
useful.

======================================================================
26. CONTENT SAFETY AGAINST HALLUCINATION
======================================================================

All AI-visible world entities should come from authoritative registries
or generated-and-persisted run-start catalogues.

If a model mentions an unknown: - NPC; - place; - item; - business; -
event; - quest; - fact;

the parser/validator must reject or strip the state-changing
implication.

Generated prose may occasionally contain harmless decorative language,
but no unknown entity becomes real merely because it was mentioned.

For consequential generation, prefer IDs over names.

======================================================================
27. DETERMINISM AND REPRODUCIBILITY
======================================================================

Not every playthrough should be deterministic, but debugging must be
reproducible enough to investigate failures.

Where possible record: - random seeds; - candidate sets; - chosen IDs; -
model/task version; - prompt template version; - accepted state delta.

Tests should use deterministic seeds and mock/fixed LLM outputs.

Do not make the test suite depend on a live LLM service.

Live-model evaluation should be a separate test layer.

======================================================================
28. FEATURE FLAGS AND STAGED ROLLOUT
======================================================================

Large experimental systems should be introduced behind developer
configuration or feature flags when appropriate.

Examples: - new Town Mind planner; - conversation consequence
extraction; - generated residents; - async scheduler; - adaptive
horror; - new memory retrieval.

A feature flag is not an excuse to leave broken code indefinitely. It
enables comparison, testing, and safe migration.

======================================================================
29. BACKWARD-COMPATIBLE EXPANSION
======================================================================

Bellwether is intended to grow: - larger village; - more areas; - more
jobs; - deeper economy; - farming; - woods; - quarry; - caves; - more
residents; - audio; - richer LLM systems.

Design current structures so reasonable expansion does not require
replacing fundamentals.

Examples: - locations should use registries/graphs rather than giant
switch statements; - NPC simulation should support tiers; - economy
should support additional businesses; - assets should use manifests; -
event primitives should be extensible; - prompts should be role-routed.

Do not prematurely build a giant generic engine, but avoid obvious dead
ends.

======================================================================
30. FAILURE MUST LEAVE THE GAME PLAYABLE
======================================================================

Unless a legitimate game failure mode has occurred, subsystem failure
should not make the game unplayable.

Examples: - LLM offline -> deterministic fallback; - optional portrait
missing -> fallback portrait/layout; - ambient audio missing -> continue
silently; - invalid procedural proposal -> reject and continue; - stale
async proposal -> discard/replan; - optional event content invalid ->
skip event and log.

Core save corruption, migration failure, or invariant violation should
be surfaced clearly rather than hidden.

======================================================================
31. DESIGN DOCUMENT PRECEDENCE
======================================================================

When sources disagree, use this order of reasoning:

1.  Current accepted working behaviour must not be accidentally
    regressed.
2.  Explicit latest user direction overrides older design intention.
3.  Canonical World and Story Bible governs story truth.
4.  LLM Role Specification governs AI authority and architecture.
5.  Engineering Context Snapshot describes current implementation.
6.  Comprehensive Project Context describes broad intended game
    direction.
7.  Older notes are historical context, not automatic authority.

If a new request intentionally changes accepted behaviour, implement it
as an explicit design change with migration and regression awareness.

======================================================================
32. DEFINITION OF DONE
======================================================================

A Bellwether feature is done when:

-   requested behaviour is implemented;
-   it integrates with authoritative state;
-   persistence implications are handled;
-   old behaviour remains working;
-   deterministic fallback exists where required;
-   LLM paths are validated where involved;
-   tests cover core rules;
-   integration tests cover affected systems;
-   UI is actually usable at runtime;
-   player-perspective behaviour has been checked;
-   no accidental debug shortcuts remain;
-   versioning is updated where appropriate;
-   complete runnable output or complete replacement files can be
    delivered.

“Code written” is not the same as “feature done.”

======================================================================
33. FINAL ENGINEERING STATEMENT
======================================================================

Bellwether should be developed as a persistent simulation, not a
sequence of disconnected features.

Its strength depends on continuity: - continuity of state; - continuity
of character; - continuity of story; - continuity of player
consequence; - continuity across failure and recurrence; - continuity of
working engineering behaviour.

The codebase should therefore favor: PRESERVATION OVER REPLACEMENT,
EXTENSION OVER REINVENTION, CAUSAL STATE OVER TEXTUAL ILLUSION,
VALIDATION OVER TRUST, MEASURED EMERGENCE OVER RANDOMNESS, COMPLETE
IMPLEMENTATION OVER DEMONSTRATION SNIPPETS, PLAYER-PERSPECTIVE TESTING
OVER TEST-SUITE COMPLACENCY.

The objective is not merely to make each new feature work in isolation.

The objective is to let Bellwether grow for years without losing the
systems, atmosphere, simulation depth, or accumulated consequences that
make the village feel alive.

13. AI PROVIDER ARCHITECTURE
======================================================================

Owner: backend/ai/provider.py AIProvider

The provider directly calls Ollama: POST {base_url}/api/generate

Default model: qwen3:1.7b

Request characteristics: - non-streaming; - keep_alive: 10m; -
configurable temperature; - configurable num_predict; - explicit
num_thread from BELLWETHER_AI_THREADS; - Qwen thinking can be disabled
with top-level “think”: false for bounded choices; - retry logic; -

recent-call memory.

Concurrency policy: Ollama is treated as a shared local resource.
AIProvider uses a condition gate: - only one generation runs at a
time; - player-facing foreground dialogue is prioritized over queued
background Directors.

This is important for low-end hardware.

Provider capabilities: - health - ask_value - ask_choice -
ask_player_reply - ask_text - compact overview routing - recent decision
routing - tracing

Bounded Director choice flow: 1. deterministic code creates legal
candidates; 2. candidates are shuffled to reduce first-option bias; 3.
compact context is attached; 4. model is asked to return exactly one
option letter; 5. output is parsed; 6. invalid/empty output returns
None; 7. deterministic fallback remains available.

This is the correct general pattern for Bellwether: GENERATE LEGAL
OPTIONS -> ASK LLM TO CHOOSE/SUGGEST -> VALIDATE -> APPLY.

Do not use: LLM -> arbitrary JSON state mutation.

BELLWETHER v0.3.0 — PART 2 VISUAL DEVELOPMENT BRIEF

Version: Planning Draft 1 Purpose: Visual Style Exploration, Asset

1. PURPOSE

Part 2 establishes Bellwether’s coherent visual identity and the
production system for future visual assets.

Bellwether remains a text-first, state-driven village life RPG. Visuals
support prose, place, character presence, emotional attachment, seasonal
familiarity, and later supernatural contradiction. The game should not
become a conventional visual novel in which images replace simulation or
prose.

The visual direction is:

-   warm pastoral life-sim at the beginning;
-   rural English village identity with a relatively modern but
    temporally ambiguous 21st-century feeling;
-   anime and visual-novel character presentation;
-   some Stardew Valley influence in warmth, seasonal attachment,
    recurring places, and readable visual identity;
-   restrained Disco Elysium influence in psychology, subjective
    perception, investigation, and later visual instability;
-   gradual movement from comforting, vibrant normality toward darker,
    stranger, less trustworthy presentation.

The goal of Part 2 is not mass asset generation. The goal is to
establish a repeatable visual language before scaling production.

2. DEVELOPMENT SEQUENCE

Part 2 should proceed in this order:

1.  Visual style exploration.
2.  Selection of the final Bellwether art direction.
3.  Asset Bible creation.
4.  Canonical character sheets.
5.  Canonical location sheets.
6.  Controlled vertical-slice asset production.
7.  Scene-stage integration.
8.  Character sprite and portrait integration.

Potential strength: Best balance of place, warmth, VN presentation, and
later atmospheric horror.

Direction B — Clean Modern Visual Novel

Characteristics: - cleaner linework; - sharper character rendering; -
simpler environmental painting; - clearer sprites and portraits; -
stronger dialogue-focused presentation.

Potential strength: Highly readable character interaction and efficient
expression production.

Potential weakness: May make the village feel more like a VN backdrop
than a living simulated place.

Direction C — Storybook Anime Life-Sim

Characteristics: - softer shapes; - slightly stylized proportions; -
warm colours; - approachable environments; - strong seasonal charm; -
gentler ordinary-life presentation.

Potential strength: Makes early Bellwether highly inviting and increases
contrast when horror corrupts it.

Potential weakness: Must avoid becoming visually childish or overly
whimsical.

Direction D — Textured Psychological Rural Illustration

Characteristics: - anime-influenced characters; - more textured and
subjective backgrounds; - selective painterly distortion; - stronger
Disco Elysium influence; - unusual lighting and psychological
atmosphere.

Potential strength: Excellent for psychology and horror.

Potential weakness: May weaken the required comforting ordinary-life
baseline if used too strongly from the beginning.

The preferred final direction may combine elements, but it must be
defined clearly enough to reproduce consistently.

Current provisional recommendation: Direction A as the foundation, with
selected warmth from Direction C and restrained psychological texture
from Direction D only when state justifies it.

BELLWETHER v0.3.0 — PART 2 UI COLOUR, ATMOSPHERE, AND VISUAL STATE DIRECTION

Version: Planning Draft 1 Purpose: Replace the transitional v0.3.0 Part
1 palette with a state-responsive visual identity

1. STATUS OF THE PART 1 COLOUR SCHEME

The v0.3.0 Part 1 colour scheme is transitional.

Part 1 focused on frontend structure and interaction grammar: - scene
stage; - contextual action categories; - shallow action trays; - quick
actions; - compact contextual rail; - dedicated information panels; -
responsive layout; - Developer Console separation.

The current dark green-grey palette should not be treated as
Bellwether’s final visual identity.

The present interface is too uniformly dark and desaturated for ordinary
opening play. A Day 1 summer or spring morning should not visually
resemble late-evening horror. Beginning the game in a permanently sombre
interface weakens the intended contrast between comforting normality and
later supernatural corruption.

Part 2 must therefore replace the fixed dark-dashboard appearance with a
state-responsive presentation system derived from the selected art
direction and Asset Bible.

2. CORE COLOUR PRINCIPLE

Bellwether should not have one static theme.

The interface should respond coherently to: - season; - time of day; -
weather; - indoor or outdoor context; - location character; -
supernatural pressure; - player psychological state where appropriate; -
recurrence state where appropriate.

The visual system should establish ordinary warmth and familiarity
before progressively allowing darkness, desaturation, contradiction, and
instability.

The interface must remain readable at all times.

3. ORDINARY DAYTIME BASELINE

Ordinary daytime play should feel welcoming, rural, calm, and alive.

Recommended UI materials: - warm cream; - soft parchment; - muted
natural green; - sunlit wood tones; - weathered stone neutrals; - gentle
earth browns; - restrained floral or seasonal accents; - dark readable
ink-like text.

Dark surfaces may still be used selectively for prose readability or
strong visual focus, but the entire interface should not sit inside
near-black panels during a pleasant morning.

The opening hours of Bellwether should visually communicate: -
possibility; - comfort; - curiosity; - rural calm; - attachment; -
ordinary life.

4. SEASONAL PALETTE BEHAVIOUR

Spring

Visual qualities: - soft fresh greens; - cream; - pale yellow warmth; -
damp earth; - blossom accents; - gentle blue-grey rain states.

Spring should feel newly active and slightly changeable.

Summer

Visual qualities: - fuller greens; - warm cream; - sunlit straw; -
deeper vegetation; - brighter sky influence; - warmer wood and stone.

Summer should feel abundant, comfortable, and socially active without
becoming oversaturated.

Autumn

Visual qualities: - ochre; - russet; - muted gold; - damp green; - warm
grey; - darkening earth.

Autumn should remain beautiful and inviting before later seasonal unease
becomes justified.

Winter

Visual qualities: - pale blue-grey; - cream; - bare-earth brown; -
subdued vegetation; - low warm interior light; - restrained cold
contrast.

Winter should feel quiet and physically demanding, but not automatically
supernatural.

5. TIME-OF-DAY BEHAVIOUR

Time should alter the interface gradually rather than switch between
unrelated themes.

Morning

-   clean warm light;
-   high legibility;
-   gentle contrast;
-   sense of possibility.

Midday

-   clearest colour identity;
-   strongest ordinary-life warmth;
-   neutral readable surfaces.

Late afternoon

-   warmer highlights;
-   lengthening shadows;
-   slightly richer earth tones.

Evening

-   reduced brightness;
-   warm artificial light where appropriate;
-   stronger separation between interior safety and exterior
    uncertainty.

Night

-   genuinely darker surfaces;
-   restrained cool environmental tones;
-   warm pools of local light;
-   stronger scene focus;
-   reduced ambient visual noise.

Night should feel different because the world is dark, not because the
game always uses a horror palette.

6. WEATHER PALETTE BEHAVIOUR

Clear

-   normal seasonal palette;
-   strong local colour;
-   clean scene readability.

Soft Overcast

-   slightly reduced contrast;
-   softer shadows;
-   retained warmth.

Overcast weather must not automatically look ominous.

Light Rain

-   cooler environmental treatment;
-   wet-surface highlights;
-   softened distance;
-   warm interface surfaces retained where appropriate.

Heavy Rain

-   stronger environmental cooling;
-   reduced distant clarity;
-   darker sky influence;
-   more pronounced shelter/interior contrast.

Snow

-   pale reflected light;
-   softened environmental contrast;
-   colder exterior surfaces;
-   warm interior accents;
-   visible snow accumulation in authored variants where available.

Storm

-   reduced saturation;
-   stronger contrast changes;
-   wind and rain treatment;
-   occasional restrained lightning;
-   darker environmental presentation.

Storm presentation should remain readable and should not resemble
supernatural corruption unless anomaly state explicitly justifies it.

Mist

-   reduced depth contrast;
-   softened distant elements;
-   restrained colour range;
-   preserved foreground readability.

7. PROSE AND DIALOGUE SURFACES

The current near-black prose box is too severe for ordinary daytime
play.

Part 2 should test alternatives such as: - translucent warm reading
surface; - parchment-like restrained panel; - lightly frosted
scene-integrated surface; - VN-style dialogue frame; -
location-responsive but consistently readable text surface.

The prose panel should remain visually central because Bellwether is
text-first.

Narration and dialogue may use related but distinct treatments.

Ordinary narration: - integrated reading surface; - calm typography; -
strong readability.

Conversation: - VN-style speaker composition; - character portrait or
bust; - speaker name treatment; - response choices clearly separated
from general actions.

Major authored scenes: - intensified version of the same visual
grammar; - stronger staging; - fewer surrounding controls; - larger
character or scene emphasis where appropriate.

8. BUTTON AND ACTION VISUAL LANGUAGE

The Part 1 action structure is retained, but the buttons require a final
visual identity.

Distinct states should exist for:

Primary action categories

People · Activities · Explore · Investigate · Travel

These should read as navigation categories rather than generic web
buttons.

Contextual actions

Examples: - Speak to Mara; - Water Garden; - Inspect the Garden; - Walk
to the Village Green.

These should be clean, readable, and connected visually to their
category.

Quick actions

Frequently used, urgent, or immediately relevant actions may surface
directly.

Examples: - Talk to present NPC; - Treat injury; - Seek shelter; - Begin

Quick actions should be visually prominent without looking like

Dialogue responses

These should have their own VN-compatible presentation and should not

Disabled actions

Dangerous or terminal actions

These require careful emphasis but should not use generic bright-red

The final UI palette should be derived from the selected Part 2 visual

Process: 1. Generate the approved style exploration set. 2. Select the

corruption tint.

Exact colour values should be chosen only after the visual style
direction is approved.

11. HORROR COLOUR CORRUPTION

Horror should corrupt an established normal palette rather than merely
make the screen darker.

Possible state-driven changes: - warmth draining from one part of the
interface; - a familiar accent colour shifting slightly; - one panel
retaining the wrong seasonal palette; - weather tint disagreeing with
the scene; - a portrait rendered with subtly incorrect tonal balance; -
a location card using impossible lighting; - recurrence annotations
using a colour not otherwise present in the current run.

Early corruption must be subtle and deniable.

Later corruption may become stronger, but should remain tied to
authoritative anomaly, psychology, or recurrence state.

Random visual glitches are not acceptable substitutes for systemic
horror.

12. MOBILE AND DESKTOP REQUIREMENTS

The colour system must work equally well on desktop and mobile.

Test: - bright screen conditions; - low-light play; - small text
surfaces; - dialogue portraits; - action trays; - scene cards; -
dedicated panels; - weather overlays; - night mode; - snow brightness; -
storm darkness.

Mobile must not simply inherit desktop transparency values if they
reduce readability over cropped artwork.

13. PART 2 IMPLEMENTATION RULE

Do not spend development time polishing the current transitional palette
as if it were final.

Instead: 1. preserve the successful Part 1 layout architecture; 2.
complete the visual style exploration; 3. choose the final art
direction; 4. create the Asset Bible; 5. derive the UI palette from the
selected artwork; 6. implement state-responsive colour tokens; 7.
integrate the first scene cards and characters; 8. visually test
ordinary life before testing horror states.

14. SUCCESS CRITERIA

The Part 2 colour and atmosphere system succeeds if:

-   a spring morning feels welcoming;
-   a summer afternoon feels alive;
-   rain feels atmospheric rather than automatically frightening;
-   snow visibly changes the world;
-   night feels naturally darker;
-   interiors can feel safe and warm;
-   storms feel physically disruptive;
-   horror corruption is distinguishable from ordinary weather;
-   prose remains easy to read;
-   illustrations and UI feel visually unified;
-   mobile and desktop both remain coherent;
-   later darkness is more powerful because the opening was genuinely
    warm.

CLOSING PRINCIPLE

Bellwether should not look haunted before the player has learned to love
it.

The opening visual language should invite the player into ordinary life.

Season, weather, and time should make the same place feel alive in
different ways.

Darkness should arrive because night falls.

Coldness should arrive because winter comes.

Unease should arrive because something familiar has become wrong.

The visual system should first teach the player what Bellwether looks
and feels like when it is normal.

Only then should Bellwether begin to change the rules.

BELLWETHER POST-v1.0 DESIGN DIRECTION AND DEVELOPMENT CONTEXT
================================================================
Status: Canonical planning context derived from the post-v1.0 discussion.
Purpose: Give ChatGPT/LLMs enough context to continue Bellwether development without requiring the user to repeat these decisions.
Scope: UI/UX, visual progression, living-world simulation, ecology, economy, LLM architecture, long-term life simulation, horror pressure, failure, side content, and post-v1.0 priorities.

IMPORTANT DEVELOPMENT RULE
--------------------------
Bellwether already has many accepted and working systems. Future work must be additive and conservative. Existing working features, simulation systems, story systems, recurrence, horror, relationships, investigation, economy, LLM architecture, Developer/Settings tools, save compatibility, and UI capabilities must not be removed, rewritten, or silently replaced without explicit consultation. Every release must be exhaustively regression-tested.

CORE CURRENT DIAGNOSIS
----------------------
Bellwether has more systems conceptually and structurally than it visibly expresses during play. The next development phase should focus on:
1. Making the village simulation genuinely connected and perceptible.
2. Activating intended but apparently dormant or incomplete runtime layers such as world_runtime and ecology.
3. Improving LLM scheduling, prompt size, latency, observability, and domain coverage.
4. Redesigning action presentation so the player is not overwhelmed by too many simultaneous buttons/options.
5. Improving visual progression from idyllic rural life toward increasingly disturbing horror.
6. Deepening life simulation without turning the game into a spreadsheet or an overly detailed Sims-like needs simulator.
7. Preserving the central philosophy: deterministic systems enforce reality and legality; the LLM acts as the village's adaptive brain and environmental/social director within strict constraints.

CURRENT RUNTIME ISSUES TO ADDRESS
---------------------------------
Observed developer output has shown:
- director_status can be AI-assisted while world_runtime remains {}.
- ecology remains {}.
- Active LLM jobs have been observed for NPC, traffic, and conversation domains.
- A simple NPC bounded decision used roughly 1,000 evaluated prompt tokens and took about 34 seconds on the low-end target system.
- A simple foreground greeting took about 60 seconds total because it waited behind background inference and then performed another large prompt evaluation.
- The async system uses a coarse director_batch coalescing key, so unrelated work may be coalesced while another domain is running.
- Prompt evaluation is the dominant performance bottleneck, not output generation.
- Developer diagnostics need to make per-domain AI activity immediately visible.

These issues must be included in future roadmap work.

UI AND PLAYER ATTENTION
-----------------------
The intended visual attention hierarchy is:
1. The background scene should remain a major focus.
2. The narrative area should show only very fresh happenings and active conversations.
3. Narration should be visually distinct from dialogue/main conversational text so it does not compete for attention. Different color, typography, framing, opacity, or presentation style may be used.
4. Current quest/thread information is important and should remain easy to see.
5. People present should use little screen space and should not compete with the scene, conversation, or current objective.
6. The action interface is currently too confusing and presents too many simultaneous options. This is a major redesign priority.

ACTION PRESENTATION PHILOSOPHY
------------------------------
Do not show every theoretically legal action at once.

The interface should use progressive contextual disclosure:
- Present actions that are currently relevant and understandable.
- Remove or clear one-time actions after they are completed.
- Unlock or reveal logical follow-up actions after prerequisites are satisfied.
- Do not randomly lock options merely to simplify the screen.
- Preserve player autonomy: progressive disclosure is an interface/progression mechanism, not arbitrary railroading.
- Prefer contextual interaction over a giant flat button list.
- NPC interaction can use a compact interaction panel opened from the NPC list, with portrait, description, and actions such as Talk, Ask, Give, Invite, Help, Observe where appropriate and legal.
- The final interaction pattern should be selected based on usability testing, but should strongly reduce button clutter.

CURRENT LOCATION PRESENTATION
-----------------------------
The location description should not be duplicated.
Keep the location title/description presentation on the scene image.
Remove the duplicate descriptive presentation from the left-side location card or repurpose that space.

PEOPLE PRESENT
--------------
Use a compact list in the normal exploration view.
Portraits should not dominate the general location screen.
When actively conversing or interacting with an NPC, show the portrait and richer NPC presentation.
An NPC interaction panel may include:
- portrait
- name
- concise player-known description
- current visible activity
- relationship impression if appropriate
- legal contextual interaction options

RECENT EVENTS
-------------
The exact final treatment is open to design judgment.
Recommendation: only genuinely fresh or important happenings should remain visible in the main interface. Older events should move into a collapsible history/event log. Repeated unchanged state should not be narrated as a new event.

BOTTOM NAVIGATION
-----------------
The exact grouping may be redesigned.
Journal, Map, Inventory, Relationships, Notebook, and Home should remain accessible, but they need not all retain identical permanent visual weight.
Any consolidation must preserve conceptual distinctions, especially between Journal and Investigation Notebook if they serve different purposes.

KEYBOARD SHORTCUTS
------------------
Keyboard shortcuts are acceptable, but they must not interfere with free-form dialogue input.
Shortcuts should only fire when focus is not inside a text input, textarea, editable field, or dialogue composer.
Mobile users must never depend on keyboard shortcuts.

MOBILE SUPPORT
--------------
Bellwether should be genuinely mobile compatible.
Do not merely shrink the desktop layout.
Use responsive composition:
- scene remains visually important
- sidebars can collapse into drawers/sheets/tabs
- contextual actions remain easy to reach
- dialogue input remains usable with the on-screen keyboard
- map and modal interfaces must fit narrow screens
- touch targets must be appropriately sized

VISUAL DIRECTION
----------------
Ordinary Bellwether should begin bright, vivid, warm, inviting, and pastoral.
As the game progresses, the visual presentation should shift gradually and subtly toward horror.

The progression should be smooth rather than a sudden horror-mode switch.
Possible controlled changes:
- reduced warmth
- altered contrast
- slightly wrong seasonal color
- subtle desaturation
- strange light direction
- fog behavior
- changed window lighting
- increasingly uneasy framing
- rare controlled interface contradictions
- portrait changes
- map inconsistencies
- text/journal contradictions
- severe but rare presentation corruption at high horror states

The contrast between early comfort and later horror is important.

SCENE VARIANTS
--------------
Long-term scene presentation should support:
- time of day: morning, afternoon, evening, night
- weather: normal/clear, rain, snow where seasonally valid, fog, and other carefully bounded states
- seasonal visual differences
- horror-state visual variants or controlled transforms

Asset growth must be managed carefully. Use a practical combination of authored variants, overlays, color/lighting treatment, and selective alternate assets rather than an uncontrolled combinatorial explosion.

SUBTLE VISUAL MOVEMENT
----------------------
Allowed if restrained and performant.
Possible examples:
- rain
- drifting fog
- chimney smoke
- subtle foliage movement
- occasional birds
- distant vehicle movement
- window light changes

Animation must not distract from text, reduce accessibility, or harm performance on low-end hardware. Reduced-motion support is required.

INTERFACE HORROR
----------------
Interface Horror may eventually become more severe at rare, high-intensity moments, including controlled layout unreliability or DDLC-like presentation disruption.
However:
- authoritative game state must never be corrupted
- accessibility must be preserved
- effects must be validator-gated and state-driven
- effects must not become random glitch spam
- Developer tools must reveal underlying truth
- the player must always be able to recover normal usability outside intended sequences

ECOLOGY
-------
Ecology should become a real persistent simulation layer.
It should generally be inferred by the player through:
- location descriptions
- Look Around/Observe actions
- gardening and farming results
- NPC remarks
- animal sightings
- shop stock and prices
- jobs and errands
- story opportunities
- environmental anomalies

Avoid turning ecology into a dashboard of raw simulation numbers in the normal player UI.

Recommended persistent ecological tendencies include:
- vegetation growth
- flowering
- fruiting
- plant stress
- soil moisture/condition
- pollinator activity
- bird activity
- fox activity
- rabbit pressure
- deer pressure
- pests
- unusual but biologically plausible seasonal deviations
- supernatural ecological anomalies only when canonical eligibility allows them

Do not simulate every individual wild animal.
Use population/activity tendencies and derive sightings/events from season, time, weather, habitat, location, recent history, and anomaly state.

ENVIRONMENTAL CAUSALITY
-----------------------
The world should have causal chains, for example:

weather history
-> soil and vegetation
-> crop/garden outcomes
-> animal activity
-> shop supply and prices
-> business pressure
-> jobs and errands
-> NPC routine changes
-> encounters and conversations
-> relationships and information flow
-> new opportunities or horror pressure

Environmental tendencies should persist for multiple in-game days. The LLM should not reconsider the whole environment from scratch every pulse.

WORLD_RUNTIME
-------------
world_runtime should become an active persistent bridge between high-level strategic direction and individual simulation systems.

The exact schema should be designed around the existing codebase, but it should meaningfully represent concepts such as:
- village mood
- village focus
- current pressures
- active tendencies
- social tension
- economic pressure
- environmental tendency
- visitor pressure
- unusual activity
- active opportunities
- pacing pressure
- quiet/active periods
- recent major changes

Do not create redundant parallel state if equivalent authoritative state already exists elsewhere. Audit first, then integrate.

ECONOMY AND BUSINESSES
----------------------
Ecology and weather should have economic consequences.
Examples:
- prolonged rain harms or delays crops
- local supply changes
- shop stock and prices change
- farms need labor
- NPC routines change
- businesses gain or lose pressure

Businesses may have persistent financial conditions such as:
- comfortable
- stable
- strained
- endangered
- recovering

Businesses may potentially close, change ownership, or be replaced, but such changes must be carefully designed so they do not casually soft-lock the game.
Economic decline may contribute to failure conditions.
The player may sometimes mitigate decline through money, work, relationships, organization, or other preparation, but not every world outcome must always be preventable.

BACKGROUND RESIDENTS
--------------------
The village can include approximately 20-25 lightweight background residents in addition to authored main and semi-main NPCs.
They may participate in bounded long-term village change such as:
- jobs
- job loss
- friendships
- conflicts
- relationships
- marriages where appropriate
- moving in or away
- local groups
- business effects
- social information flow

Their simulation must remain lightweight and should not consume the same LLM/context budget as main NPCs.

BIRTHS AND DEATHS
-----------------
Allowed in very long games, but rare and carefully bounded.
Respect realistic timescales and biology.
Avoid absurd demographic churn caused by compressed game time.

LLM ROLE AND PHILOSOPHY
-----------------------
Bellwether's LLM role must remain central.
Optimization must not turn Bellwether into a mostly deterministic game with occasional AI flavor text.

The LLM is intended to be:
- a constrained brain of the village
- a social and environmental driver
- a strategic pacing intelligence
- a chooser among legal possibilities
- a generator of bounded dialogue and selected side content
- a system that observes player patterns and adapts pressure
- part of the mechanism that makes different playthroughs diverge

The deterministic engine remains authoritative over:
- legality
- canonical facts
- story gates
- geography
- inventory truth
- relationship arithmetic/state
- time
- persistence
- validation
- safety boundaries
- application of proposals

LAYERED LLM COGNITION
---------------------
A layered approach is acceptable and recommended as long as it preserves the game philosophy:

1. Deterministic systems construct legal possibilities and compact relevant state.
2. Small bounded LLM prompts choose among meaningful alternatives.
3. Deeper cognition/strategy runs less frequently and updates persistent goals, tendencies, plans, pressures, or interpretations.
4. Later small decisions use those persistent strategic results rather than resending the entire history every time.

This should reduce latency without reducing meaningful AI authority.

FOREGROUND AND BACKGROUND PRIORITY
----------------------------------
Direct player conversation is foreground work and must receive strong priority.
Because inference on the target hardware is non-preemptive, scheduling should avoid launching expensive low-priority work when a foreground interaction is likely.

Background work should be scheduled intelligently during:
- sleep
- longer travel
- work shifts
- menus/reading where appropriate
- player inactivity
- other natural simulation windows

Do not run simultaneous large model inference jobs on the 8 GB low-end target merely to claim concurrency.
Keep one inference active at a time if memory/CPU contention makes concurrency harmful, but maintain a useful prioritized queue.

ASYNC SCHEDULER IMPROVEMENTS
----------------------------
Replace coarse global coalescing with domain-aware scheduling or a pending-domain accumulator.

Unrelated work should not disappear because another domain is active.
For example, NPC inference should not erase pending ecology work.

Potential priorities:
1. foreground player conversation
2. urgent player-facing NPC response
3. major event resolution
4. urgent NPC decision
5. Town Mind strategic work
6. social arc work
7. ecology/environment
8. economy/business
9. traffic/visitors
10. ambient conversation
11. memory compression/reflection

Exact priority rules should be profiled and tested.

PROMPT OPTIMIZATION
-------------------
Current prompts are too large for many bounded tasks.
A simple NPC action choice should not routinely require around 1,000 evaluated prompt tokens.

For bounded NPC decisions, include only task-relevant compact state such as:
- identity/voice essentials
- current location
- current activity
- current dominant need or intent
- active obligation
- immediate relevant memory
- nearby relevant actors
- weather only if relevant
- recent action IDs for repetition avoidance
- legal choices

Deep cognition should be updated less frequently and stored as structured persistent state.

Foreground dialogue should retrieve only relevant memories and immediate story constraints rather than sending broad cognition and world state indiscriminately.

DEVELOPER AI OBSERVABILITY
--------------------------
The Developer Console should expose AI activity by domain.

For every domain show:
- implemented/disabled/dormant state
- last requested time
- last started time
- last success
- last fallback
- current queued/running/completed-waiting state
- model
- prompt token/evaluation count
- inference duration
- queue wait
- accepted count
- rejected count
- stale count
- timeout/failure count
- coalesced count
- next eligibility
- current persistent tendency/result where appropriate

Example:
NPC: last success 09:45 | 34s | 1041 prompt tokens | AI applied
Ecology: pending | no successful run yet
Weather: last success Day 1 06:00
Town Mind: next eligible Day 3

The Developer Console should make it possible to prove whether each intended AI role actually runs.

FARMING
-------
Long-term farming should be somewhat simpler than Stardew Valley, but not a one-click plant-and-harvest system.

Potential depth:
- seasonal crop suitability
- soil condition
- moisture
- weather influence
- crop health
- basic quality variation
- simple fertilizer/soil improvement
- pests/disease in restrained form
- harvesting
- cooking/preservation connections
- selling and market effects
- larger agricultural land as expensive late-game progression
- possible farm animals later if practical

The LLM should influence bounded crop/environmental tendencies and unusual conditions, while deterministic rules enforce biology, timing, legality, and calculations.

JOBS
----
Use a hybrid model:
- scheduled recurring jobs/work shifts
- flexible jobs
- errands
- side-quest-like temporary work
- dynamically emerging opportunities

Work can consume in-game time and may occasionally contain decisions, social encounters, complications, or events.
Jobs can be lost and new opportunities can emerge dynamically.

BUSINESS OWNERSHIP
------------------
The player may eventually be able to own or operate a business.
This should be a long-term progression system, not immediate early-game access.
It should connect to economy, relationships, jobs, supply, ecology, and village change.

COOKING
-------
Exact depth is open to design judgment.
Recommended middle depth:
- ingredients
- recipes
- basic quality/outcome variation
- experimentation in bounded form
- food as health/energy/recovery support
- NPC preferences and gifts
- social/event uses
- limited selling of prepared food
Avoid excessive micromanagement.

HOBBIES AND SOCIAL COMMUNITIES
------------------------------
Hobbies should create social connections and recurring communities.
Examples:
- fishing -> recurring anglers and waterside encounters
- gardening -> growers, Mara, exchanges, seasonal concerns
- local history -> Ruth, Mrs Ellis, archives, inscriptions
- cooking -> bakery/shop/social events
- walking/nature -> ecology observations and encounters

Repeated activity should improve relevant skills naturally and unlock opportunities.

LLM-GENERATED SIDE CONTENT
--------------------------
Side quests and social situations may continue to appear indefinitely.
The LLM may inspect overall authoritative game state and propose where to apply pressure or opportunity next.

Generated side content must:
- use existing world facts
- respect geography and NPC identity
- respect economy and current state
- be validated
- avoid inventing canon-breaking lore
- avoid pretending generated content is an authored central revelation
- persist legally once accepted
- have bounded objectives and resolution conditions

HORROR PRESSURE
---------------
Horror must not be entirely missable.

The player may choose to focus heavily on farming, work, relationships, hobbies, or comfort, but the adaptive system should observe playstyle and eventually introduce appropriate horror pressure.

If the player becomes too comfortable in life-sim routines, the Town Mind/LLM should respond within canonical constraints.
This does not mean arbitrary punishment every time the player relaxes.
The pressure should be paced, psychologically effective, and connected to existing systems.

The game should preserve comfort and curiosity so that horror has something meaningful to threaten.

FAILURE
-------
Logical failure modes may extend beyond ordinary injury/death.
Possible families include:
- physical death
- severe injury/setback
- psychological collapse
- social exile or isolation
- financial ruin
- institutional confinement
- incorporation/identity loss where canonically appropriate
- world/economic cascade failures
- other authored or validator-approved outcomes

Some negative world developments may occur even when the player did not directly cause them.
The player may mitigate some through preparation, skills, relationships, money, health, knowledge, or intervention.
Not every adverse outcome must always be preventable.

Failure must connect coherently to recurrence/new-run architecture.
The activation and presentation of new runs must be audited and clarified:
- what constitutes terminal failure
- when recovery is possible
- when recurrence is offered or triggered
- what persists
- what resets
- how the town remembers
- how the next run changes

POST-ENDING SUPERNATURAL STATE
------------------------------
Exact behavior may vary by ending.
Recommendation:
- some endings make Bellwether substantially safer
- some contain or redirect the threat
- some create unstable peace
- some preserve danger
- some change the player's relationship with the town

Postgame should remain playable and reflect the achieved ending.

SIDE MYSTERIES AFTER AUTHORED CONTENT
-------------------------------------
LLM-generated side quests may continue after authored side mysteries are exhausted.
However, generated supernatural content must remain within canonical ontology and must not casually invent major new lore that contradicts the authored mystery structure.

SAVE, RELOAD, AND NEW RUNS
--------------------------
Continue maturing:
- clear New Game vs Continue vs Load vs browser refresh semantics
- browser refresh should reconnect to current server state
- explicit confirmed fresh reset
- versioned save metadata
- migration
- backup recovery
- corruption handling
- provenance
- safe interaction with async jobs
- cancellation/rejection of stale jobs after load/reset
- clear recurrence/new-run transitions

PERFORMANCE TARGET
------------------
Primary low-end reference:
- Intel i3-4160
- 8 GB RAM
- single-channel memory
- approximately 4 GB zram swap
- Debian headless
- no discrete GPU

The game should remain responsive during deterministic play even if Ollama inference takes 60 seconds.
Only direct free-form conversation may normally require visible waiting.
Background AI must continue to matter and must not be silently removed for performance.

Testing should include deliberately delayed inference and verify:
- movement remains responsive
- gardening remains responsive
- jobs remain responsive
- inventory remains responsive
- map remains responsive
- menus remain responsive
- save/load/reset remain race-safe
- background results are later validated and applied
- foreground dialogue receives appropriate priority

HOLISTIC DESIGN PRINCIPLE
-------------------------
Bellwether should not grow by endlessly adding disconnected systems.
The next development era should make existing systems interact visibly.

The player should feel:
- the village changes
- NPCs have lives
- weather matters
- ecology matters
- businesses react
- routines change
- information travels
- relationships have consequences
- the town observes player habits
- horror responds intelligently
- failures have history
- new runs remember asymmetrically
- ordinary life remains worth returning to

The desired result is an emergent village simulation in which authored story, deterministic simulation, and constrained LLM direction reinforce each other.
