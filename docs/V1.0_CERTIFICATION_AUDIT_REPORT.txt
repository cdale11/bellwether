BELLWETHER v1.0 CERTIFICATION AUDIT REPORT

Scope
-----
Certification pass over the accepted v1.0 RC3 baseline. No baseline file was removed. The pass focused on UI decluttering, action presentation, save/reload/reset semantics, language and repetition cleanup, long-session memory bounds, async/state race surfaces, story/progression regression, and distribution integrity.

Implemented corrections
-----------------------
1. Main utility UI decluttered: Save, Load, and History moved into Menu; Developer / Settings remains directly accessible.
2. Immediate/quick actions are no longer duplicated in category trays.
3. Removed the generic display fallback “The village keeps its own quiet rhythm.” Authored location descriptions are now used when weather has no special effect.
4. Recent-event display suppresses duplicate speaker/text entries.
5. Engine-side conservative consecutive deduplication added for identical player-facing messages and world events.
6. Added explicit confirmed Reset to Fresh Game. Browser reload semantics are documented: reload reconnects to the running server state; it does not reset the game.
7. Save system now writes provenance metadata, validates temporary JSON, keeps one last-good backup, and can recover from that backup if the primary save is damaged.
8. Long-session bounds reinforced for encounters, AI event history, and per-NPC social memory. Existing bounded provider traces and async lifecycle event logs were verified.
9. /api/state now acquires the shared game lock because view generation performs migration/refresh work and is not a purely passive read.
10. README rewritten in simpler language. Roadmap copies corrected to preserve the agreed v0.9.0 recurrence -> v0.9.1 failure/recovery sequence and updated with v1.0 certification status.

Known limits
------------
This certification pass improves repetition control conservatively. It does not attempt semantic paraphrase detection across all prose, because aggressive fuzzy suppression could hide genuinely meaningful repeated facts. Live Ollama latency and memory behaviour remain hardware-dependent and should still be observed on the target i3-4160 system during long sessions.
