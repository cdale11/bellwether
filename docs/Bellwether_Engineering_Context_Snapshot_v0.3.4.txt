====================================================================== 1.
ENGINEERING OPERATING RULE
======================================================================

INSPECT FIRST. EXTEND SECOND.

Bellwether already contains substantial working simulation, narrative,
AI, persistence, horror, recurrence, economy, activity, UI, and
diagnostic architecture.

Do not: - casually rewrite backend/core/game.py; - replace working
models with new parallel systems; - create duplicate geography,
relationship, memory, investigation, horror, economy, job, recurrence,
or AI truth; - let LLM output directly mutate arbitrary state; - remove
deterministic fallback behaviour; - break old-save migration; - infer
implementation solely from README text, because some root documentation
is stale relative to the current packaged build.

Before modifying a subsystem: 1. inspect the owning model and Game
integration; 2. inspect its authored content catalogue; 3. inspect its
diagnostic; 4. inspect frontend rendering if player-facing; 5. preserve
save compatibility; 6. run cumulative and targeted diagnostics; 7.
perform player-perspective playtesting for meaningful changes.

13. AI PROVIDER ARCHITECTURE
======================================================================

Owner: backend/ai/provider.py AIProvider

The provider directly calls Ollama: POST {base_url}/api/generate

Default model: qwen3:1.7b

Request characteristics: - non-streaming; - keep_alive: 10m; -
configurable temperature; - configurable num_predict; - explicit
num_thread from BELLWETHER_AI_THREADS; - Qwen thinking can be disabled
with top-level “think”: false for bounded choices; - retry logic; -
timeout handling; - adaptive token-budget recovery for empty
length-exhausted output; - debug traces; - health state; - compact
recent-call memory.

Concurrency policy: Ollama is treated as a shared local resource.
AIProvider uses a condition gate: - only one generation runs at a
time; - player-facing foreground dialogue is prioritized over queued
background Directors.

This is important for low-end hardware.

Provider capabilities: - health - ask_value - ask_choice -
ask_player_reply - ask_text - compact overview routing - recent decision
routing - tracing

Bounded Director choice flow: 1. deterministic code creates legal
candidates; 2. candidates are shuffled to reduce first-option bias; 3.
compact context is attached; 4. model is asked to return exactly one
option letter; 5. output is parsed; 6. invalid/empty output returns
None; 7. deterministic fallback remains available.

This is the correct general pattern for Bellwether: GENERATE LEGAL
OPTIONS -> ASK LLM TO CHOOSE/SUGGEST -> VALIDATE -> APPLY.

Do not use: LLM -> arbitrary JSON state mutation.

29. SAVE, LOAD, MIGRATION, AND COMPATIBILITY
======================================================================

Save file: saves/save.json

Save behaviour: - compile LLM overview; - serialize complete state
JSON; - write sibling temporary file; - parse temporary JSON to
validate; - atomically replace save.json.

Load behaviour: - parse save JSON; - reset overview cache; - run
migrate_state; - rebuild provider recent-call memory from saved session
summary; - move message cursor to recent scene rather than replaying
full lifetime transcript; - recompile overview.

Migration: Game.migrate_state contains broad setdefault/default-merging
behaviour across old and new domains.

Subsystem models also expose migration/default mechanisms where
relevant.

Rule for future state: Every new persistent state domain must have: -
INITIAL_STATE default; - migration/defaulting for old saves; -
JSON-serializable values; - round-trip test; - targeted old-save test.

38. CURRENT ARCHITECTURAL RISKS / TECHNICAL DEBT

A. GAME.PY CENTRALIZATION backend/core/game.py is a very large
orchestration file with many responsibilities. Do not rewrite it, but
future features should avoid making it even more monolithic when a
focused model can own domain logic.

B. STATE LAYER OVERLAP There are legacy and structured state domains
that coexist. This is not automatically wrong, but future changes must
trace actual reads/writes before consolidation.

C. DOCUMENTATION VERSION DRIFT README heading is stale relative to
VERSION and current code.

D. VISUAL DIAGNOSTIC FAILURE Part 20 currently fails “approved art
packaged” in the inspected build.

E. ASSET SIZE Current PNG scene/portrait assets are several MB each. The
future Asset Bible should define WebP/PNG policy and compression targets
for distribution builds.

F. CORE CAST SCALE Current deep authored cast is only three NPCs. Vision
target is roughly 6–7 core NPCs plus about 20 lighter residents.
Expansion should preserve knowledge/social simulation constraints.

G. RECURRENCE VISION GAP Current recurrence is a bounded new-run
carry-forward system. The newer vision calls for richer persistence: -
completed story missions persist; - context-sensitive recovery
locations; - mixture of literal and uncanny continuity; - progressive
memory return. This should be an extension of RecurrenceModel and story
persistence, not a replacement.

H. UI ASSET MANIFEST / CHARACTER MISMATCH Packaged portrait folders
include characters beyond the three current core-cast catalogue entries.
Future asset cleanup should distinguish active canonical NPCs, planned
NPCs, and unused assets.

42. ONE-PARAGRAPH TECHNICAL SUMMARY
======================================================================

Bellwether is a FastAPI + vanilla-JS stateful simulation game whose
authoritative runtime is orchestrated by a large Game class and a set of
deterministic domain models backed by authored JSON catalogues. The
local Ollama model acts through a constrained AIProvider and Director
architecture: legal candidates and compact context are constructed by
code, the LLM chooses or generates bounded output, results are parsed
and validated, and deterministic fallbacks preserve playability. The
game already contains world routing, autonomous NPC purpose, social
relationships, knowledge propagation, free-text conversation, gardening,
economy, jobs, events, seasons, investigation, systemic horror, evolving
player identity, danger, death, recurrence, cooking, home restoration,
persistent saves, old-save migration, an API-driven UI, visual assets,
and a persistent exploration map. Future work should extend these
substrates conservatively, preserve authored truth and knowledge
boundaries, keep the LLM constrained, and validate changes with both
diagnostics and real player-perspective playtesting.
